Nacos 核心 ​

最新版本：Nacos 3.2.x（2026年），AI原生 + AP/CP双模增强。

概述：Nacos（Dynamic Naming and Configuration Service）是阿里巴巴开源的动态服务发现、配置管理和服务管理平台。Nacos 的名字来源于 Naming + Configuration + Service。

知识图谱 ​

入门级：基础概念 ​

Nacos 是什么 ​

Nacos 是阿里巴巴在 2018 年开源的云原生中间件，定位为动态服务发现、配置管理和服务管理平台。它集合了 Eureka（服务注册发现）、Consul（服务健康检查）和 Apollo（配置管理）的核心能力，以"一个平台解决多个问题"的理念降低微服务架构中的组件碎片化问题。

Nacos 的核心设计哲学：

  • 服务是核心：围绕"服务"这一微服务架构的一等公民，提供全生命周期管理。
  • 动态是关键：支持实时感知服务状态变化，实现配置的动态刷新而不重启应用。
  • 云原生优先：从架构设计之初就考虑了容器化、Kubernetes 等云原生场景的适配。

Nacos 在 2023 年毕业成为 CNCF 顶级项目，截至 2026 年已在全球超过 10 万家企业中部署使用。

典型应用场景 ​

服务发现流程 ​

Nacos 支持两种模式：

  • UDP 推送（本机）：服务端变更时通过 UDP 广播通知同机器上的客户端
  • HTTP 轮询：客户端定时（默认 10s）向服务端拉取最新的服务实例列表

核心原理深度解析 ​

1. Distro 协议详解 ​

Distro 协议是 Nacos 自研的最终一致性分布式协议，专门为服务发现的 AP 场景设计。

                    ┌─────────────────────────────────────┐
                    │         Distro 协议架构              │
                    └─────────────────────────────────────┘

      Node-A ◄──────── Peer-to-Peer (对等架构) ────────► Node-B
        │                                                      │
        ├── 接收写请求     (任意节点可写)                        ├── 接收写请求
        ├── 同步到其他节点 (异步 + 重试)                        ├── 同步到其他节点
        ├── 版本号对账     (定期全量/增量)                       ├── 版本号对账
        └── 健康检查去重   (按节点分摊)                          └── 健康检查去重

五大核心机制：

  1. 对等无 Leader 架构：所有节点地位对等，任意节点都可接收客户端的写请求。对比 Raft 协议：Raft 所有写操作必须经过 Leader，Leader 宕机时需要选举，期间无法写入。

  2. 写操作流程：

    客户端 → Node-X → 写入本地数据存储 → 立即返回成功（不等同步结果）
                  → 异步发送 DistroTask 给其余节点
                  → 目标节点写入本地并返回 ACK
                  → 若同步失败，写入重试队列，延迟重试
    1
    2
    3
    4

    关键点：先写本地，异步同步。这意味着写操作延迟约等于单机写入延迟。

  3. 异步同步机制：同步任务以 DistroTask 形式放入任务队列，默认重试 3 次，每次间隔递增。

  4. 版本号对账机制：每个服务实例数据携带一个单调递增的版本号。节点间定期（默认 5s）进行全量对账 + 增量对账。

  5. 健康检查去重：Nacos 按 hash(serviceName) % clusterSize 将健康检查任务分摊到不同节点。

Distro vs Raft 对比 ​

对比维度DistroRaft
一致性级别最终一致强一致
写入方式任意节点可写，异步同步必须经过 Leader，同步提交
写入延迟低（本地写即返回）较高（需多数节点确认）
Leader 依赖无 Leader，对等架构依赖 Leader，选举时不可用
数据对账版本号驱动的定期对账Raft Log 天然保证
适用场景服务注册发现配置管理

为什么 Distro 没有采用 Raft？ ​

  1. 写入必须经过 Leader：如果 Leader 所在节点网络故障，在选举完成前整个集群都无法写入。
  2. 选举延迟：Leader 选举通常需要 150ms~300ms，在此期间服务注册请求会失败。
  3. 跨地域部署困难：Raft 要求多数节点确认写入，跨地域部署时延迟会显著增加。
  4. 复杂度高：Raft 的状态机、日志压缩、快照等机制增加了运维难度。

3. 配置动态刷新原理 ​

Nacos 的配置动态刷新是一个三层协同机制：

    ┌─────────────────────────────────────────────────────────────┐
    │                 Nacos 配置动态刷新三层模型                     │
    └─────────────────────────────────────────────────────────────┘

    第一层：长轮询 (Long Polling)           ← 传输通道
    客户端发起 HTTP 请求 → 服务端挂起 30s → 有变更返回 / 超时重来

    第二层：@RefreshScope                 ← 对象生命周期管理
    Spring Cloud 提供的注解，标记的 Bean 在配置变更时会被重新创建
    原理：CGLIB 代理 → 缓存失效 → 下次访问时重建 Bean

    第三层：事件驱动 (RefreshEvent)         ← 自定义扩展
    配置变更后发布 RefreshScopeRefreshedEvent 事件
    应用可通过 @EventListener 监听并执行自定义逻辑

长轮询 vs 短轮询 vs WebSocket：

方案实时性资源消耗实现复杂度防火墙友好

Nacos 选择长轮询的核心原因：兼容性好、资源效率高、实现简单。

第二层：@RefreshScope 原理 ​

  1. 代理包装：被注解的 Bean 会被 CGLIB 生成代理对象。
  2. 懒加载缓存：代理对象内部维护一个 ObjectFactory，首次访问时创建真实 Bean 并缓存。
  3. 缓存失效：监听到 RefreshEvent 时，调用 ScopeCache.clear() 清空缓存。
  4. 延迟重建：下一次用户请求访问该 Bean 时，代理重新创建新 Bean，注入最新配置值。

注意

@RefreshScope 仅对 @Value 注入的属性生效。@ConfigurationProperties 绑定的类无需 @RefreshScope，Spring Boot 内置机制会自动处理。

经典高频面试题 ​

Q1：Nacos 的 AP 模式和 CP 模式有什么区别？如何选择？ ​

答案：

对比维度AP 模式CP 模式
一致性协议自研 Distro 协议（最终一致）Raft 协议（强一致）
实例类型临时实例（ephemeral=true，默认）持久实例（ephemeral=false）
Leader无 Leader，对等架构，任意节点可读写有 Leader，写操作必须经 Leader 提交
写入一致性最终一致性（秒级收敛）强一致性（写入即一致）
网络分区表现分区两端都可独立提供服务少数派分区不能写入、不能选主
健康检查客户端心跳上报，超时自动删除服务端主动探测，仅标记状态

从请求路径理解两种模式：

AP 模式（Distro）的写操作路径：

客户端 → 任意 Nacos 节点 → 本地写入 → 立即返回成功 → 异步同步到其他节点
1

核心思想：先保证可用性，后追求一致性。

CP 模式（Raft）的写操作路径：

客户端 → Nacos Leader 节点 → 同步到 Follower → 多数节点确认 → 返回成功
1

核心思想：先保证一致性，后考虑可用性。

选型指南：

  1. 优先选择 AP 模式：微服务节点弹性扩缩容、Serverless、网络环境不稳定场景
  2. 选择 CP 模式：DNS-F 模式服务解析、核心网关路由表、金融级服务治理
  3. 混合使用：同一集群中部分服务用临时实例（AP），部分用持久实例（CP）

Q2：Nacos 配置中心如何实现配置动态刷新？ ​

答案：

Nacos 配置动态刷新的核心是一个三层协同机制：长轮询（传输通道）→ @RefreshScope（Bean 生命周期）→ 事件驱动（自定义扩展）。

第一层：长轮询（Long Polling）：客户端发起 HTTP 请求携带当前配置的 MD5 列表，服务端比对 MD5，有变更立即返回，无变更则挂起 30s。使用 Servlet 3.0 AsyncContext 挂起请求而不阻塞线程。

第二层：@RefreshScope：CGLIB 代理包装 Bean → 懒加载缓存 → 收到 RefreshEvent 时清空缓存 → 下次访问时重建 Bean 并注入最新配置值。

第三层：事件驱动：

事件时机用途
RefreshEventBean 被标记为 dirty触发 RefreshScope 清除缓存
RefreshScopeRefreshedEvent所有 RefreshScope Bean 刷新完成重建连接池、清空本地缓存
EnvironmentChangeEventEnvironment 层面属性变更监控配置项变更并告警

Q3：Nacos 如何防止服务注册中的假死问题？ ​

答案：

Nacos 的防假死机制：

1. 临时实例：分阶段心跳处理，避免误判

  客户端心跳发送 (5s) → 服务端状态判断:
         
  上次心跳距现在 < 15s            → HEALTHY (健康)
  上次心跳距现在 15s ~ 30s        → UNHEALTHY (不健康, 但保留)
  上次心跳距现在 > 30s            → 剔除实例
1
2
3
4
5

Nacos 引入了 15s 的"不健康但不剔除"缓冲区：

  • 即使连续 3 次心跳丢失（最多 15s），实例也不会被立即删除。
  • 只有持续 30s 无心跳才会剔除。
  • 在"不健康"状态期间，如果心跳恢复，实例会立即变回 HEALTHY。

2. 持久实例：仅标记状态，从不自动删除

Nacos vs Eureka 假死处理对比：

维度NacosEureka
核心机制分层超时（不健康缓冲区 15s + 剔除 30s）自我保护模式 + 心跳超时
自我保护无专门的自我保护模式核心特色功能
心跳超时30s 无心跳剔除默认 90s 无心跳剔除
误剔除风险低（缓冲区 + 用户可配置）中（自我保护模式正确配置下较低）

为什么 Nacos 没有自我保护模式？

  • Nacos 的临时实例已经通过 15s 缓冲区 + 30s 剔除的超时设计，在网络短暂抖动时不会误剔除。
  • Nacos 的 AP 模式采用 Distro 协议，网络分区两端可独立工作，分区恢复后通过版本号对账自动合并数据。
  • Nacos 认为避免大规模误剔除的责任应该在机制设计层面解决，而非引入一个"关闭剔除"的开关。

Q4：Distro 协议是如何工作的？ ​

答案：

Distro 是 Nacos 自研的最终一致性 AP 协议，核心思想：对等无 Leader 架构 + 异步同步 + 版本号对账。

完整写操作流程：

步骤1: 客户端向 Node-A 发起服务注册请求
步骤2: Node-A 执行本地写入（更新内存注册表 + 更新版本号）
步骤3: Node-A 立即返回成功给客户端（不等待同步结果）
步骤4: 异步同步阶段
         ├── Node-A 构造 DistroTask
         ├── 放入同步队列 → 向 Node-B 发送 HTTP 请求
         ├── Node-B 收到后检查版本号 → 更新本地注册表
         ├── 同步失败 → 进入重试队列 (默认 3 次)
         └── 重试仍失败 → 进入 Dead Queue

   仅对版本号有差异的服务进行数据同步，避免全量数据传输

Q5：Nacos 和 Eureka 的区别是什么？ ​

答案：

维度NacosEureka
开发厂商阿里巴巴Netflix
开源状态活跃维护（3.x 2026年）2.0 停止开发（2018年停更）
核心功能服务发现 + 配置中心 + 服务治理仅服务注册发现
CAP 模型AP/CP 双模式可切换仅 AP
一致性协议Distro(AP) + Raft(CP)Peer to Peer(AP)
健康检查客户端心跳(临时) + 服务端探测(持久)客户端心跳
配置管理原生内置、功能完善无
实例类型临时实例 + 持久实例只有一种实例类型
Kubernetes3.x 原生 Operator + xDS 协议需 Sidecar 适配
社区活跃度高（CNCF 项目）低（停止维护）

核心差异：

  1. 功能定位：Eureka 是纯服务注册发现，Nacos 是"三合一"（服务发现 + 配置管理 + 服务治理）。
  2. 一致性模型：Eureka 纯 AP，Nacos AP/CP 可切换。
  3. 健康检查：Eureka 只支持客户端心跳，Nacos 支持心跳 + 服务端探测。

迁移建议：新项目首选 Nacos。已有的 Eureka 项目可以平滑迁移到 Nacos（Nacos 提供 Eureka 兼容 API）。

Q6：Nacos 3.x 相比 2.x 有哪些核心升级？ ​

答案：

Nacos 3.x（3.0 于 2025 年发布，3.2.x 于 2026 年发布）是一次里程碑式的架构升级，核心升级方向聚焦在五个方面：

1. AI 原生（AI Native）：智能路由、异常检测、容量预测、自然语言交互、智能配置推荐、根因分析。

2. 性能大幅提升：

指标Nacos 2.xNacos 3.x提升倍数
最大实例数~10万~100万+10x+
注册 TPS~5,000~50,00010x
配置推送延迟1~3s<500ms3~6x
内存占用(万实例)~2GB~500MB4x

3. Kubernetes 原生：官方 Operator、xDS 协议支持、ConfigMap 双向同步、Sidecar 自动注入。

4. 安全增强：mTLS 全面支持、命名空间级 RBAC、完整审计链路、AES-256 加密。

5. 可观测性：OpenTelemetry Traces、Prometheus Metrics、Grafana Dashboard、集群健康度评分。

升级建议：Nacos 2.x 可以直接升级到 3.x（兼容 2.x 的 API 和数据格式）。建议先在非核心环境验证 2 周。风险点：gRPC 长连接模型变化可能导致现有网络策略需要调整。
