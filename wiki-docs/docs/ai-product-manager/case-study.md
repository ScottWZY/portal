# 实战案例：AI 智能客服从 0 到 1 全流程

## 概述

本章以一个**真实的电商 AI 智能客服项目**为线索，完整呈现 AI PM 从需求验证到上线迭代的全流程工作。每个阶段都附有实际的交付物片段和决策依据——不是理论推演，而是"如果让你重做一遍，你会怎么做"级别的还原。

::: tip 为什么选智能客服？
智能客服是 AI PM 最经典的入门场景——技术成熟度高、ROI 清晰、A/B 测试容易、数据闭环完整。同时也是面试中最有可能被问到的"你做过什么 AI 产品"的场景。读完本章你就能在面试中讲一个完整的端到端案例。
:::

## 项目背景

- **产品**：某电商平台的智能客服系统
- **现状**：日均 5000 通在线咨询，50 人客服团队，高峰期排队 > 5 分钟
- **目标**：AI 自动处理 40% 的咨询量，人工转接率降低 30%
- **时间线**：10 周从立项到全量上线

---

## 一、需求验证阶段（第 1-2 周）

### 1.1 第一步：用 AI 适用性矩阵判断

不是所有需求都适合用 AI 解决。我做的第一件事是用五维评估矩阵打分：

| 维度 | 评估 | 得分 | 说明 |
|------|------|------|------|
| **数据可行性** | 高 | 3 | 有 2 年历史对话数据，覆盖 80%+ 咨询场景 |
| **技术可行性** | 高 | 3 | GPT-4o API + RAG 架构 POC 已跑通 |
| **商业价值** | 高 | 3 | 预计年省 200 万人力成本，ROI > 500% |
| **用户体验** | 高 | 3 | 用户真实痛点——等待时间太长 |
| **合规风险** | 中等 | 2 | 需做数据脱敏 + 用户知情同意 |

**总分 14 分 → P0 优先级，干。**

### 1.2 第二步：快速 POC 验证可行性

用 3 天时间搭了一个极简 Demo：

- **架构**：GPT-4o API + 100 条历史客服 QA 做 RAG 知识库
- **测试方法**：拿出 50 条真实用户咨询，让 AI 回答 + 人工评分
- **POC 结果**：

| 指标 | POC 结果 | 上线最低标准 |
|------|---------|-------------|
| 回答准确率 | 78% | 85% |
| 用户满意度评分 | 3.2 / 5 | 4.0 / 5 |
| 平均响应时间 | 2.1 秒 | < 3 秒 |

**结论**：78% 的准确率虽然不够上线，但方向成立。差距主要来自 RAG 检索不准（知识库文档写得不够结构化）和 Prompt 太粗糙。

### 1.3 第三步：定义上线标准

在产品还没有的情况下先定义"什么叫成功"，这是 AI PM 最重要的习惯：

- **技术指标**：准确率 ≥ 85%，召回率 ≥ 80%
- **业务指标**：AI 自动解决率 ≥ 40%，人工转接率下降 ≥ 30%
- **体验指标**：用户满意度 ≥ 4.0/5，响应时间 < 3 秒
- **上线决策**：灰度 5% 流量跑一周，三项都达标再扩量

## 二、MVP 设计阶段（第 3-4 周）

### 2.1 定义 AI 的能力边界

这是最容易被 PM 跳过的步骤——很多人一上来就开始写 Prompt，结果做了三个月发现方向错了。

我定义了三层能力边界：

```
┌────────────────────────────────────────────┐
│ L1: AI 自主处理（目标 40% 咨询量）            │
│ • 退货流程咨询 • 物流查询 • 优惠券规则         │
│ • 常见商品问题 • 账号操作指导                  │
└────────────────────────────────────────────┘
┌────────────────────────────────────────────┐
│ L2: AI 辅助 + 人工确认（目标 25% 咨询量）       │
│ • 复杂退换货（多商品多订单）                    │
│ • 金额相关的投诉（需确认金额）                   │
└────────────────────────────────────────────┘
┌────────────────────────────────────────────┐
│ L3: 直接转人工（目标 35% 咨询量）              │
│ • 情绪激动用户（检测到负面情绪关键词）           │
│ • 涉及赔偿/举报/法律风险                       │
│ • AI 置信度 < 80%                            │
└────────────────────────────────────────────┘
```

### 2.2 设计三级降级策略

AI 产品的降级设计比正常流程设计更重要——因为模型一定会犯错。

| 场景 | 降级策略 | 触发条件 |
|------|---------|---------|
| **模型高置信度（> 95%）** | 直接返回 AI 回答 | 正常 |
| **模型中置信度（80-95%）** | 返回知识库原文 + 标注"以下信息来自帮助中心" | 兜底 |
| **模型低置信度（< 80%）** | 转人工 + 提示"正在为您转接人工客服" | 止损 |

### 2.3 MVP 的功能范围

| 包含（MVP） | 不包含（V2） |
|------------|-------------|
| 文本问答 | 语音/图片问答 |
| 退货、物流、优惠券三大场景 | 全品类商品咨询 |
| RAG 知识库检索 | 基于用户画像的个性化推荐 |
| 满意/不满意反馈按钮 | 多轮对话深度交互 |
| 转人工兜底 | 工单自动创建 |

## 三、数据准备阶段（第 3-5 周）

### 3.1 标注规范设计

这是 AI PM 的核心交付物之一。我为三大场景各设计了一份标注规范。以"退货退款"场景为例：

::: details 退货退款场景标注规范（节选）

**标签体系**

| 标签 | 定义 | 示例 |
|------|------|------|
| `return_policy` | 询问退货政策/流程 | "怎么退货？""七天无理由怎么算？" |
| `return_progress` | 查询退货进度 | "我的退款什么时候到？""退货快递收到了吗？" |
| `return_exception` | 退货异常/投诉 | "退货被拒绝了""为什么扣了我的运费？" |
| `exchange_request` | 换货请求 | "想换个颜色""尺码不合适换大一号" |
| `other` | 以上都不属于 | "这个商品什么时候补货？" |

**边界 Case 定义**

| 用户输入 | 正确标签 | 容易误标为 | 判别规则 |
|----------|---------|-----------|---------|
| "退货太慢了，我要投诉" | `return_exception` | `return_progress` | 有投诉意图优先归 exception |
| "退了再买一个，能便宜吗" | `return_policy` | `other` | 退货意图明确，归 policy |
| "怎么退不了啊？？？" | `return_exception` | `return_policy` | 包含操作受阻，归 exception |
| "收到了，谢谢" | `other` | - | 无退货相关意图 |

**质量控制标准**
- Kappa 系数 ≥ 0.75（两人标注一致性）
- 黄金集准确率 ≥ 95%（100 条标准答案校准）
:::

### 3.2 首批数据标注

| 环节 | 数量 | 成本 | 周期 |
|------|------|------|------|
| 意图分类（5 类 × 3 场景） | 3000 条 | ¥900（0.3 元/条） | 1 周 |
| 问答对（RAG 知识库） | 500 对 | ¥500（内部标注） | 1 周 |
| 边界 Case 专项补充 | 200 条 | ¥100（内部标注） | 2 天 |
| **合计** | **3700 条** | **¥1,500** | **2 周** |

### 3.3 数据 Pipeline 搭建

```
用户咨询 → 实时日志采集 → Kafka → 数据清洗（脱敏+去重+格式化）
                                        ↓
                                  标注任务分发（LabelStudio）
                                        ↓
                                  标注审核（Kappa 检验）
                                        ↓
                                  训练/评测数据集（版本管理 v1.0）
```

## 四、模型方案选型（第 4-5 周）

### 4.1 方案对比

| 方案 | POC 效果 | 成本/次 | 延迟 | 迭代难度 |
|------|---------|---------|------|---------|
| **A: GPT-4o API + 基础 Prompt** | 准确率 78%，用户体验一般 | ¥0.05 | ~1.5s | 低（改 Prompt 即可） |
| **B: GPT-4o API + RAG + 优化 Prompt** | 准确率 85-88%，回答有依据 | ¥0.08 | ~1.8s | 中（改 Prompt + 知识库） |
| **C: 方案 B + 精调小模型** | 准确率 90-93%，成本更低 | ¥0.02 | ~0.3s | 高（需重训） |

**选择方案 B 作为 MVP，方案 C 作为 V2 目标。**

理由：
- 方案 B 能在 10 周内上线（方案 C 需要额外 6-8 周精调）
- 先用方案 B 积累真实数据，为方案 C 提供高质量训练集
- 大模型当老师，小模型当打工人——业界性价比最优路径

### 4.2 成本预估

| 项目 | 月度成本 | 说明 |
|------|---------|------|
| GPT-4o API（意图分类 + 答案生成） | ¥1,500 | 日均 5000 次 × 30 天 × ¥0.01（仅意图分类用大模型） |
| Embedding 检索 | ¥200 | 知识库检索 |
| 精调小模型推理（V2 阶段） | ¥300 | 意图分类下沉到自部署小模型后大幅降本 |
| 知识库维护人力 | ¥5,000 | 1/4 人力，持续维护 |
| **合计（V1）** | **¥6,700/月** | vs 20 个人工客服 ¥167,000/月 |

## 五、Prompt 迭代优化（第 5-6 周）

### 5.1 版本演进

这是整个项目中最"AI PM"的工作——持续调 Prompt，看 Bad Case，再调 Prompt。

**V1（第 5 周初，准确率 82%）**

```
你是电商客服助手。请根据用户问题给出回答。
```

问题：模型经常答非所问，有时编造不存在的退货政策。

**V2（第 5 周末，准确率 88%）**

```
## 角色
你是 XX 电商的智能客服助手。友好、专业。

## 知识范围
只能基于【参考资料】回答问题。
如果参考资料中没有答案，必须说"建议您联系人工客服获取准确信息"。
严禁编造任何信息。

## 参考资料
{检索到的帮助中心文档}
```

改进点：加了知识范围约束和 RAG 检索。准确率提升 6 个点。但发现模型在"参考资料里有但是表述不同"的场景下还是会编。

**V3（第 6 周末，准确率 93%）**

在 V2 基础上增加了：
- Few-shot 示例（3 个正确回答 + 2 个"不知道"的正确示范）
- 输出格式约束（先给简短答案，再给"I 帮助中心参考链接"）
- 敏感词触发规则（检测到"投诉/举报/赔偿"→强制转人工）

::: tip 迭代经验
每一次 Prompt 迭代都应该回答三个问题：
1. 这版改了哪些 Bad Case？（不是感觉，是具体哪类 Case）
2. 有没有引入新的问题？（V3 的"不知道"示范导致模型在某些本该能回答的场景也说不知道——矫枉过正了）
3. 下一步的优化假设是什么？
:::

### 5.2 Bad Case 分析——AI PM 最重要的日常

每周五下午固定流程：

1. 从上一周的线上日志中抽取 100 条 AI 回答
2. 人工标注每个回答的质量：✅ 好 / ⚠️ 可接受 / ❌ 差
3. 对 ❌ 的 Case 分类归因

| 错误类型 | 占比 | 根因 | 解决方案 |
|----------|------|------|---------|
| 检索到的文档不对 | 35% | 知识库文档标题不匹配用户表达 | 优化文档标题 + 查询改写 |
| 答案是错的 | 25% | 模型在"不够确定"时也给答案 | 提高"不知道"触发的置信度阈值 |
| 答案太啰嗦 | 20% | Prompt 没约束输出长度 | 加"回答不超过 3 句话" |
| 转人工太激进 | 20% | 敏感词检测过于宽泛 | 精准化敏感词列表 |

## 六、灰度上线与 A/B 测试（第 7-9 周）

### 6.1 灰度策略

| 阶段 | 流量比例 | 持续天数 | 放量条件 |
|------|---------|---------|---------|
| 内部灰度 | 仅内部账号 | 3 天 | 无明显 Bug |
| 小流量 | 5% 用户 | 5 天 | 核心指标不劣于对照组 |
| 中流量 | 20% 用户 | 5 天 | 继续观察稳定性 |
| 全量 | 100% | 持续 | — |

### 6.2 A/B 测试实验设计

- **实验组**：用户咨询时先由 AI 回答，不满意转人工
- **对照组**：保持纯人工客服
- **样本量计算**：预期人工转接率从 95% 降到 60%，α=0.05，power=0.8，每组需要约 800 次咨询。5% 流量 3 天能跑够。

### 6.3 A/B 测试结果（小流量 5% × 5 天）

| 指标 | 对照组 | 实验组 | 变化 | 显著性 |
|------|--------|--------|------|--------|
| 人工转接率 | 95.0% | 57.2% | **-39.8%** | p < 0.001 ✅ |
| AI 自动解决率 | — | 42.8% | — | — |
| 用户满意度 | 4.1 | 3.9 | -0.2 | p = 0.08（不显著） |
| 平均响应时间 | 120s | 2.3s | **-98%** | p < 0.001 ✅ |
| 客服人效（咨询/人/天） | 100 | 175 | **+75%** | p < 0.001 ✅ |

**结论**：核心指标全面正向，用户满意度轻微下降但不显著（可能是对新交互模式的不适应）。通过放量评审，进入中流量阶段。

### 6.4 灰度中踩的坑

1. **"谢谢"和"好的"被 AI 当成新问题**：用户说"谢谢"之后 AI 又开始重复回答。修复：增加对话终结识别。
2. **高峰期 API 超时**：晚 8-10 点 GPT-4o API 延迟从 1.5s 飙升到 8s。修复：加超时降级（>5s 直接转知识库兜底回答）。
3. **Android 端表情包异常**：用户发微信自带表情包时，API 收到的是乱码。修复：预处理阶段过滤非文本 Unicode 字符。

## 七、数据闭环与持续优化（第 10 周起）

### 7.1 数据漂移监控

上线第 3 周，发现 AI 自动解决率从 43% 降到了 38%。排查流程：

1. **确认不是波动**：连续 5 天下降，统计检验显著 → 是真的下降了
2. **看数据分布**：双十一大促开始后，"退货退款"类咨询占比从 30% 飙到 55%
3. **根因定位**：退货场景的复杂 Case（多商品多订单）比例从 10% 升到 25%，这些 Case 准确率只有 72%
4. **解决**：紧急补充 200 条退货复杂场景的 QA 对到知识库 + 微调意图分类模型 → 自动解决率恢复到 41%

### 7.2 V2 阶段的三个优化方向

| 优化方向 | 方法 | 预期收益 | 优先级 |
|----------|------|---------|--------|
| **意图分类下沉到精调小模型** | 用积累的 5000 条真实标注数据微调 BERT → 意图分类推理成本降低 95% | 月省 ¥1,200 | P0 |
| **多轮对话能力** | 引入对话状态追踪——记住用户上一轮说的是哪个订单、什么商品 | 用户满意度 +0.3 | P1 |
| **个性化推荐** | 基于用户画像和购买历史，推荐相关帮助文档 | 自动解决率 +5% | P2 |

## 八、面试追问合集

### Q1: 这个项目中你最大的决策失误是什么？

::: details 答案

我选方案 B（GPT-4o + RAG）是对的，但低估了 RAG 检索质量对产品体验的影响。

最初我用了最简单的向量检索——Embedding + 余弦相似度。上线后发现一个尴尬的问题：用户搜"退货"时，检索到的文档是"如何发起退货"（正确），但用户搜"退款"时，由于"退款"的操作入口在"退货"流程里，检索返回的是"如何查看退款进度"——但用户其实还不知道怎么退款。

这是典型的"用户表达 ≠ 系统文档标题"问题。我犯了所有 AI PM 第一次做 RAG 都会犯的错误——**高估了向量检索的能力，低估了查询改写的必要性**。

后来加了查询改写模块：用户输入"我的退款什么时候到"→ 改写为"退货进度查询+退款到账时间"→ 再用改写后的查询去检索。准确率提了 8 个点。

如果重来一次，我会在 POC 阶段就做"查询改写对检索质量的影响"专项测试，而不是等到上线后才补救。
:::

### Q2: 你怎么说服管理层投入资源做这个项目？

::: details 答案

我没有讲技术方案，直接算了一笔经济账：

"我们 50 个客服，保守估计 40% 的咨询量是标准化问题（退货流程、物流查询、优惠券规则），这些不需要人类的判断力。如果 AI 能处理这 40%，相当于省掉 20 个客服的人力。20 个客服的年成本是 200 万。AI 方案的成本是多少？GPT-4o API 每月 1500 块，加上检索、维护，每月不到 7000 块，一年 8 万。200 万 vs 8 万——这是一笔不需要犹豫的投资。"

然后我补了一句："最重要的是，我们不是花 200 万赌 AI 能不能行——我们可以先用 5 万块做 POC，两周出 Demo。Demo 跑通了再决策要不要投全量。"

这两句话的效果：第一句给信心（ROI 高），第二句给安全感（试错成本低）。管理层当场就批了。
:::

### Q3: 上线一年后回头看，你最大的认知变化是什么？

::: details 答案

最大的认知变化是：**AI 产品的壁垒不是模型能力，是数据和流程的深度耦合**。

刚做这个项目时，我觉得"接个 GPT API + 写个 Prompt"就是 AI 产品了。一年后我明白那只是 Demo。

真正的壁垒是：
1. 我们积累了 50 万条真实对话数据 + 3 万条高质量人工标注——这个数据集的价值远超任何公开数据集，因为它完全贴合我们的业务场景。
2. 标注规范沉淀了 200+ 条边界 Case 定义——这是团队知识的结晶，新标注员一周就能上手。
3. Prompt 经历了 47 次迭代——每一次迭代都不是拍脑袋，而是基于 Bad Case 的精确分析。
4. 数据闭环跑通后——用户反馈 → Bad Case 分析 → 知识库更新 → 模型再训练 → 效果验证——这个飞轮一旦转起来，竞品拿不走。

如果我只停留在"写 Prompt"的层面，这个产品上线 3 个月就会被竞品追平。但因为我们在数据和流程上扎得足够深——一年后竞品用同样的 GPT 模型也做不出我们的效果。
:::
