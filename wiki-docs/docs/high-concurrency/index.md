# 高并发架构

## 模块概述

高并发架构是高级工程师面试中的"决胜题"，考察候选人对系统整体设计、瓶颈分析和方案选型的综合能力。本模块覆盖分层架构、扩展策略、高可用设计、分布式核心理论和经典场景方案，帮助你建立完整的系统设计思维框架。

::: tip 核心思路
高并发不是单一技术的堆砌，而是**分层架构设计** + **各层独立扩展** + **瓶颈精准识别**的系统工程。
:::

::: info 设计原则
先做垂直扩展探顶，再做水平扩展；先读写分离，再分库分表；先缓存加速，再异步解耦。
:::

## 核心知识点

### 架构分层与扩展

| 子模块 | 核心内容 |
|--------|----------|
| 分层架构 | 客户端层 → CDN → 反向代理层（Nginx） → 网关层 → 业务层 → 数据层 |
| 垂直扩展 | 升级 CPU / 内存 / 磁盘，SSD 替换 HDD，瓶颈与极限 |
| 水平扩展 | 无状态服务弹性伸缩、有状态服务的数据分片、负载均衡策略 |
| 动静分离 | 静态资源 CDN + OSS、动态页面服务化 |
| 读写分离 | 主从复制延迟处理、强制读主、ShardingSphere 读写分离配置 |

### 缓存架构

| 子模块 | 核心内容 |
|--------|----------|
| 多级缓存 | 浏览器缓存 → CDN 缓存 → Nginx 本地缓存 → Redis 分布式缓存 → 本地缓存（Caffeine） |
| 缓存预热 | 冷启动问题、定时任务预热、流量回放预热 |
| 缓存更新 | Cache Aside / Read Through / Write Through / Write Behind 模式对比 |
| 热点处理 | 热点 Key 发现（客户端采集 + 上报）、多级拆分（本地 + 分布式）、多副本 |

### 异步与削峰

| 子模块 | 核心内容 |
|--------|----------|
| 消息队列削峰 | 瞬时流量 → MQ → 平滑消费，消费速度控制 |
| 请求合并 | 高频小请求合并为批量处理（Hystrix Request Collapsing） |
| 异步处理 | CompletableFuture 编排、MQ 异步解耦、响应式编程（Reactor/WebFlux） |

### 高可用设计

| 子模块 | 核心内容 |
|--------|----------|
| 冗余与容灾 | 同城双活、两地三中心、多活架构的挑战（数据同步/路由） |
| 限流 | 计数器/滑动窗口/漏桶/令牌桶算法对比、Nginx 限流、Sentinel 控制台 |
| 降级 | 降级策略（返回默认值/缓存/兜底页面）、自动降级 vs 手动降级 |
| 熔断 | 熔断器状态机（Closed → Open → Half-Open）、慢调用比例 vs 异常比例 |
| 隔离 | 线程池隔离（Bulkhead）、信号量隔离、舱壁模式 |

### 分布式理论

| 子模块 | 核心内容 |
|--------|----------|
| CAP 定理 | 一致性、可用性、分区容错三者取舍，不同中间件的 CAP 选择 |
| BASE 理论 | 基本可用、软状态、最终一致性，AP 系统的设计哲学 |
| 一致性协议 | 2PC/3PC（XA 事务）、TCC（Try-Confirm-Cancel）、Paxos/Raft（共识算法） |
| 分布式 ID | 雪花算法（Snowflake）详解、号段模式（Leaf）、Redis 自增 |
| 分布式锁 | Redis（SET NX + Lua / RedLock / Redisson）、Zookeeper（临时顺序节点）、Etcd |

### 场景设计

| 场景 | 核心方案 |
|------|----------|
| 秒杀系统 | 前端限流 → 网关校验 → 库存预热到 Redis → Lua 原子扣减 → MQ 异步下单 |
| 短链系统 | 发号器生成唯一 ID → Base62 编码 → 302 重定向 → 布隆过滤器防穿透 |
| 排行榜 | Redis ZSet（实时榜单） + 定时任务归档到 MySQL（历史榜单） |
| 消息推送 | WebSocket 长连接 → 心跳保活 → 连接路由表 → MQ 广播 |
| 文件上传 | 分片上传 → 秒传（MD5 校验）→ OSS 直传 → CDN 加速 |

## 面试重点

::: warning 高频考点
1. **秒杀系统设计**：从前端到后端到数据的完整链路，每层的瓶颈和解决方案
2. **CAP 理论应用**：结合具体中间件说明取舍（Nacos AP / ZK CP / Eureka AP）
3. **分布式 ID**：雪花算法时钟回拨问题怎么解决？Leaf 号段模式为什么比雪花更可靠？
4. **缓存一致性**：先删缓存还是先更新数据库？延迟双删解决了什么？Canal 方案怎么玩？
5. **限流算法对比**：漏桶 vs 令牌桶 vs 滑动窗口，各自适用场景
6. **分布式事务**：2PC/3PC/TCC/Saga/可靠消息最终一致性，画图对比，选型标准
7. **系统容量估算**：给定 DAU/QPS，估算服务器数量、带宽、存储容量
:::

::: danger 容易翻车的点
- 直接将单机方案套用到分布式场景，忽视数据一致性问题
- 设计没有量化思维，不说具体数字（QPS、RT、容量）
- 方案停留在概念层，不能落地到具体技术和配置
- 只考虑功能，不考虑异常情况下的降级和容错
:::

## 学习建议

### 阶段一：理论奠基（2 周）
1. 深入理解 CAP、BASE、一致性哈希、Quorum 等分布式核心理论
2. 阅读 Raft 论文，理解分布式共识的核心思想
3. 用纸笔画出 5 个经典场景的架构图（秒杀、短链、推送、IM、Feed 流）

### 阶段二：组件实战（3 周）
4. 搭建 Redis Cluster + Sentinel，模拟主从切换和脑裂
5. 用 JMeter 压测 Nginx → Gateway → Sentinel → 业务服务的全链路
6. 实现雪花算法，模拟时钟回拨问题并给出解决方案

### 阶段三：场景模拟（2 周）
7. 完成秒杀系统的代码实现（限流 + 缓存预热 + Lua + MQ 异步下单）
8. 设计并实现一个分布式锁压测，对比 Redis vs ZK vs Etcd 性能差异
9. 编写系统设计文档模板，从需求 → 容量估算 → 架构图 → 数据流 → 异常处理全流程
