## 三、SOLID 设计原则

| 原则 | 说明 |
|------|------|
| **S** - 单一职责 | 一个类只负责一件事 |
| **O** - 开闭原则 | 对扩展开放，对修改关闭 |
| **L** - 里氏替换 | 子类可以替换父类，不影响程序正确性 |
| **I** - 接口隔离 | 不依赖不需要的接口方法 |
| **D** - 依赖倒置 | 依赖抽象而不是具体实现 |

## ⭐ 面试高频问题

### Q1：你用过哪些设计模式？分别说说使用场景？

| 模式 | 场景 | 实际案例 |
|------|------|----------|
| **单例** | 全局唯一实例 | 配置管理类、数据库连接池、Spring Bean 默认作用域 |
| **代理** | 控制对象访问，无侵入增强 | Spring AOP（日志、事务、权限）、RPC 远程调用 |
| **策略** | 算法族，避免 if-else | 支付方式切换、排序算法选择、数据校验规则 |
| **责任链** | 请求沿链传递 | 审批流程、Servlet Filter、Spring Security 过滤器链 |
| **观察者** | 事件通知 | Spring 事件机制、消息推送、GUI 事件监听 |
| **模板方法** | 定义算法骨架 | JdbcTemplate、RestTemplate、AbstractList |

### Q2：单例模式的 DCL（双重检查锁定）为什么需要 volatile？还有其他写法吗？

**为什么需要 volatile**：`new` 操作不是原子的，JVM 可能指令重排序（先分配内存 → 赋值引用 → 再初始化对象）。如果线程 A 执行到赋值引用但未初始化，线程 B 拿到未完成的对象，导致 NPE。volatile 禁止指令重排序。

**其他线程安全的单例写法**：
- **静态内部类**：利用 JVM 类加载机制，Holder 类首次被引用时才加载，天然线程安全
- **枚举**：JVM 保证线程安全，防反射和序列化破坏，是最佳实践

### Q3：为什么要面向接口编程？依赖倒置原则是什么意思？

**面向接口编程**：程序依赖于抽象（接口/抽象类），而不是具体实现。这样更换实现时不需要修改调用方代码。

**依赖倒置原则（DIP）**：高层模块不应该依赖低层模块，两者都应该依赖抽象。抽象不应该依赖细节，细节应该依赖抽象。

```java
// ❌ 依赖具体实现：换数据库需要改代码
public class UserService {
    private MySQLDao dao = new MySQLDao();  // 依赖具体类
}

// ✅ 依赖抽象：换数据库只需换实现
public class UserService {
    private UserDao dao;  // 依赖接口
    public UserService(UserDao dao) { this.dao = dao; }
}
```

### Q4：开闭原则是什么？如何实现？

**开闭原则（OCP）**：对扩展开放，对修改关闭。当需求变化时，应该通过扩展现有代码来实现，而不是修改已有代码。

**实现方式**：
- 使用**策略模式**：新增算法无需修改已有代码
- 使用**工厂模式**：新增产品无需修改工厂逻辑
- 使用**抽象类/接口**：定义扩展点，让子类实现

### Q5：设计模式用得越多越好吗？什么情况下不需要用设计模式？

**不是越多越好**。设计模式是为了解决特定问题，过度使用会导致代码复杂、难以维护。

**不需要用设计模式的情况**：
1. **需求简单明确**：写一个简单的工具类，不需要工厂模式
2. **未来不会扩展**：一次性代码，不需要为"可能的扩展"过度设计
3. **团队不熟悉**：引入团队不熟悉的模式反而增加沟通成本
4. **性能敏感**：某些模式引入了额外间接层，影响性能

**核心原则**：KISS（Keep It Simple, Stupid）和 YAGNI（You Ain't Gonna Need It）。先简单实现，确实需要时再重构引入模式。
