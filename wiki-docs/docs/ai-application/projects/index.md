## 面试高频题

### Q1: 三个实战项目的难度递进关系是什么？每个项目最核心的技术是什么？

**详细答案：** 我们当时这三个项目确实是故意从易到难设计的，刚好覆盖企业 AI 应用最常见的三类场景。我做第一个项目——知识库问答时，最深的体会是真的要在细节上下功夫，而不是堆架构。比如分块，一开始我直接按固定长度切，结果召回准确率只有 62%，后来改成语义分块+重叠滑动窗口，准确率一下子涨到 78%。这个项目比的就是谁把 RAG 全链路打磨得更精细，从文档解析到 Embedding、混合检索、Rerank，每一环都影响最终效果。我记得掌握好项目一，企业里 80% 的 AI 应用需求你都能独立接了。

第二个项目是 AI 代码助手，难度上来了，核心变化是从"检索完直接生成"的一步流程，变成了"分析→检索→生成→检查→修复"的循环。我们当时一开始没做循环，让 LLM 一次生成完，结果经常出现漏查依赖、编译错误的问题。后来加上 Agent 循环，让它自己调用编译工具检查，错了就自己改，通过率从 56% 提升到了 83%。这个项目最关键是理解 Agent 的设计思路——不是让 LLM 包办一切，而是让它学会调用工具。第三个项目智能数据分析就更综合了，要把 RAG、Agent 和结构化输出结合起来做 NL2SQL，真的就是企业里实实在在要解决的问题。

### Q2: 为什么项目一（企业知识库问答）建议先用 Dify 快速搭建，再手写代码？

**详细答案：** 我自己就是这样学的，先 Dify 后手写，感受很深。我们给市场部做产品知识库原型时，用 Dify 只花了一个下午就搭出来了，拖拽式操作，把 30 多份文档传上去，选好 Embedding 模型和向量库，立马就能问答。这个体验太重要了——几个小时你就能看到 RAG 全链路是怎么工作的，建立全局认知，比看十篇 paper 都直观。很多同事上来就手写代码，结果三天过去了还在调试文档解析，连个 demo 都没跑起来。

第二个好处是 Dify 给了你一个可量化的基线。我们用 Dify 版本跑了运营给的 120 条真实问题，准确率大概是 58%。这就给了明确目标——手写版本必须超过这个数。后来我们重构了文档解析和混合检索策略，最终做到了 74%。而且 Dify 的架构确实值得参考，它的文档处理流水线、Prompt 编排方式，我们手写时借鉴了不少。不过说实话，Dify 做原型很快，但当你需要自定义文档切分策略或者接入自己训练的 Rerank 模型时，它就很局限了，最后不得不自己写。

### Q3: 在三个项目的技术栈中，为什么选择 FastAPI + Vue3 + Element Plus 这套组合？

**详细答案：** 选 FastAPI 不是为了赶时髦，是真的解决了实际问题。我们项目里 LLM 调用动辄 3-5 秒，用同步框架比如 Flask 的话，一个慢请求就能堵住整个线程池，后面的请求全部排队。FastAPI 的 async/await 配合 httpx 异步请求，在 4 核 8G 的机器上轻松撑住了 30+ 并发，每次请求都在调 OpenAI API，这要放在同步框架上直接就挂了。还有一个巨爽的点是 Swagger 自动文档——我们接口改了 4 版参数，每次改完 Swagger UI 自动刷新，前端同学直接在浏览器里就能调接口、看返回格式，省掉了至少一半的沟通时间。

前端的 Vue3 + Element Plus 也是踩坑后的选择。我们最初试过 React + Ant Design，但团队里两个前端同学对 Hooks 不够熟练，进度拖了两周。换成 Vue3 后，Composition API 的写法他们上手很快，加上 Element Plus 组件库封装度很高，一个聊天界面+文档管理后台两天就撸完了。Vite 的热更新也是真实爽——调试 Prompt 生成效果时要频繁改 UI 文案，秒级刷新，不像 Webpack 动不动等十几秒。这套组合在国内生态已经很成熟了，中文文档全、社区活跃，招人也方便。

### Q4: 从项目一到项目三，技术架构的演进路径是怎样的？

**详细答案：** 这三个项目的架构变化，其实就是我在不同阶段遇到的真问题迭代出来的。项目一的知识库系统是最简单的单向流水线：请求进来→检索文档→拼 Prompt→LLM 回答→返回。这个阶段 90% 的精力都花在检索质量调优上了，比如我们文档分块从固定 512 tokens 改成按段落语义切分后，Retrieval 准确率从 62% 拉到了 78%。

项目二就不是一条直线了，而是 Agent 循环。我们最开始偷懒，用和项目一一样的单向架构做代码助手，结果生成出来的代码经常编译不过。后来下决心引入 Agent 循环——生成代码→调编译器检查→出错就分析错误信息→修正后重新编译，最多循环 3 轮。就这么一个改动，代码可用率从 28% 直接跳到 61%。这里有个小取舍：我们当时纠结要不要加自动 fix 循环，但技术 leader 觉得"小 bug 可以自动修，大改动还是要开发自己来"，所以设了 3 轮上限，效果刚刚好。

项目三就更复杂了，是个多引擎协作架构。我们做的是让销售 VP 用自然语言查数据库，最大的坑是 SQL 生成经常出错。于是我们加了一个校验层——生成 SQL 后先在测试库 dry-run，如果语法错误就把错误信息塞回 Prompt 让 LLM 修正，最多重试 2 次。加上这个自修复机制，SQL 生成成功率从 65% 提到了 89%。还有一个细节：遇到"环比增长"这种需要两步计算的 query，我们用两个 Agent 分工——一个写基础 SQL，另一个计算结果指标，比让单个 LLM 端到端处理靠谱得多。

### Q5: 在项目实战中，Docker Compose 部署方案解决了哪些实际问题？

**详细答案：** Docker Compose 在我们这种小团队的 AI 项目里，真的是救命级的东西。我们吃过最大的亏就是环境不一致——开发机是 Python 3.11+CUDA 12.1，测试服务器却是 Python 3.9+CUDA 11.8，ChromaDB 在两个环境下行为完全不一样，排查了整整一天才定位到是环境差异。Docker 把整个运行时环境锁死后，这三个环境再也没出过这种幺蛾子。

我们系统有至少 4 个服务：FastAPI 后端、Vue 前端、ChromaDB 向量库、还有 Redis 缓存。之前每次部署要逐个启动、检查端口占用、配网络互通，运维同学头大。Docker Compose 一个 YAML 加 docker compose up -d，一行命令 30 秒搞定。更重要的是资源隔离——我们 16G 内存的服务器，给 ChromaDB 分了 6G、后端 4G、前端只给 512M，精确控制后从来没 OOM 过。对需要私有化部署的客户来说——很多企业不愿上公有云——Docker Compose 是门槛最低的方案。我们给 3 个客户做了私有化交付，都是这套方案，运维成本基本为零。
