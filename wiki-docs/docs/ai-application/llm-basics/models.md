## 六、关键趋势观察

### 6.1 API 兼容标准化

现在 **几乎所有模型都兼容 OpenAI SDK**，只需替换 `base_url` 和 `api_key` 即可切换模型，迁移成本极低。

```python
# 统一调用代码，切换模型只需改两行
client = OpenAI(
    api_key="YOUR_API_KEY",
    base_url="https://api.xxx.com/v1"  # 切换模型服务商
)
```

### 6.2 价格战持续

DeepSeek V4-Pro 永久降价 75%，模型价格持续下探。现在最便宜的输入价格已降到 **$0.10/M tokens** 以下，比一年前便宜了一个数量级。

### 6.3 推理模型分化

o3/o4-mini（OpenAI）、DeepSeek R1（深度求索）、Claude Extended Thinking（Anthropic）等推理专用模型形成独立赛道。复杂推理任务交给专用推理模型，成本和准确率都更好。

## 七、面试高频题

### Q1: GPT-4o 和 Claude 系列如何选择？它们的核心差异在哪里？

**详细答案：**
我们项目两个模型都用过，分工很明确。GPT-4o 我们用在需要多模态的场景——用户上传商品图片问"这个能不能退货"，GPT-4o 直接看图判断商品状态，然后对接我们的退货策略回答，这个能力 Claude 目前追不上。另外 OpenAI 的生态太成熟了——function calling、structured outputs、Code Interpreter，我们搭建 Agent 几乎不用自己造轮子。

Claude 我们专门用在长文档处理场景。我们的用户协议和产品说明书都是几十页的长文档，GPT-4o 128K 窗口读到后面就开始"跑题"，而 Claude 全系标配 1M 窗口，读 30 万字的长文档中间不漏关键条款。另外 Claude 的代码和 Agent 能力是真的强——SWE-bench 得分领先，我们的代码审查辅助和自动化测试生成就全靠它。价格上 GPT-4o $2.5/$10（输入/输出百万 Token）、Claude Sonnet 4.6 $3/$15，差不太多，但 Claude 同样价格下给了 1M 窗口。

有个细节值得提：Anthropic 的安全对齐做得更激进，Claude 有时候会过度谨慎拒绝一些合理请求。我们测过一次把公司内部文档给 Claude 做分析，它看到"撤销支付"相关的内容就拒绝分析了，其实只是文档里提到了支付流程，根本不是不安全的操作。所以如果在国内某些合规约束较多的场景，GPT-4o 反而更实用。我们的策略是：核心推理用 Claude，多模态和生态工具用 GPT-4o，批量任务降级到 DeepSeek。

### Q2: 国内模型和国际模型的主要差异是什么？什么情况下应该选择国内模型？

**详细答案：**
我们的客服系统用了"国际为主、国内为辅"的混合架构，体会很深。主要差异就三个：中文能力、合规、价格。中文方面，Qwen 和 DeepSeek 确实比 GPT-4o 对中文口语和网络用语的理解更到位——用户说"这破手机烫手"，GPT-4o 可能按字面理解成"手机很热"，而 Qwen 能理解这是一种吐槽并关联到退货意图。合规这块是硬约束：我们的客服数据含用户订单号和联系方式，用国内 API 数据不出境既满足监管要求也省得做数据脱敏。

价格差距是真的大——DeepSeek V3 输入 $0.27/M、GPT-4o $2.5/M，差了 9 倍。我们日消耗 80 万 Token，如果全用 GPT-4o 一个月光 API 费用就上万美元。用 DeepSeek 接 70% 的常规流量、GPT-4o 只处理复杂问题，成本砍了六成。但国际模型在复杂推理和多模态上还是领先的——我们试过让 DeepSeek R1 和 o3 同时跑数学推理题，o3 准确率高 15 个点。

所以我们的策略是：常规客服对话用 DeepSeek V3（中文好、便宜），复杂推理和长文档分析用 Claude Sonnet（能力强），数据出境敏感的全量切 Qwen 国内节点。混合架构可能是未来很长时间的常态。

### Q3: 如何根据业务场景选择合适的大模型？请给出几个具体例子。

**详细答案：**
模型选型本质是在"能力、成本、延迟、合规"四个维度做 trade-off，没有绝对的"最好"模型，只有最适合场景的模型。通用决策框架是：先明确场景的核心约束（成本敏感？必须长上下文？需要合规？），再在满足约束的前提下选择能力最强的模型。

我们自己的典型选型：Agent 编程用 Claude Sonnet（SWE-bench 领先、工具调用稳定），高并发客服用 DeepSeek V3（中文好、便宜、QPS 能撑到 200+），复杂推理用 o3 或 R1（推理链可解释），长文档处理用 Claude Sonnet 1M 窗口（20 万字合同"大海捞针"准确率 99%），批量降本用 Gemini Flash-Lite（输入 $0.10/M，成本可忽略）。关键原则：不要一开始就上最贵模型，先解决问题再根据瓶颈升级。

### Q4: 价格因素如何影响模型选型决策？什么情况下应该追求极致性价比？

**详细答案：**
大模型 API 定价差异极大，从 $0.10/M 到 $10/M 差了 100 倍，而且输出价格通常是输入的 2-4 倍——很多人只算了输入没算输出，最后账单翻倍。我们的成本优化靠两层策略：模型降级 + 前缀缓存。模型降级效果最直接——我们分析了一周的数据，发现 70% 的用户问题其实不需要 GPT-4o 级别的回答质量，切到 DeepSeek V3 或者 Gemini Flash 完全够用，API 成本直接砍了六成。

前缀缓存也是意外的省钱利器。我们的 System Prompt 大概 500 Token，DeepSeek 缓存命中只需 $0.028/M（原价 $0.27/M），这部分几乎免费。我们把 System Prompt 和 RAG 的固定文档部分都放在消息最前面，最大化缓存收益。如果要极致性价比：批量处理用 Gemini Flash-Lite（$0.10/M）、日调用量百万级以上务必上缓存、把简单任务从旗舰模型降级到 Flash/Mini 模型。我们现在的成本结构是：DeepSeek 占 40%、Qwen 占 30%、OpenAI 占 20%、Claude 占 10%，总月费控制在 5000 块以内。

### Q5: 为什么现在所有模型都兼容 OpenAI API 格式？这带来了什么好处？

**详细答案：**
OpenAI API 格式已经是事实上的行业标准了——我们项目 4 个不同的模型供应商全是 OpenAI 兼容的，代码切模型几乎只改两行：`base_url` 和 `model`。这不是偶然，是生态推动的结果：OpenAI 是第一个做大模型商业 API 的，早期开发者都在它的 SDK 上写，后来 DeepSeek、Qwen、Mistral 这些服务商发现——我如果不兼容 OpenAI API，开发者就要为我学一套新 SDK，迁移成本会让很多人直接放弃用我的服务。所以大家都选择兼容，降低了开发者切换门槛。

这给我们带来了实实在在的好处。第一是开发效率——我们所有模型调用的代码完全一样，改配置就能切模型，不需要为每个供应商维护一套适配层。第二是容灾降级简单——主模型 A 挂了，一行配置切换到备用模型 B，对业务完全透明。如果没有标准化，写这种 fallback 逻辑要适配每家 API 的差异。第三是自由选择——我们可以随时比较不同模型的效果和价格，哪个划算用哪个，不会被锁定在某一家。

但要注意一个误区：接口兼容不等于能力兼容。DeepSeek 虽然 API 格式和 OpenAI 一样，但 `response_format`（structured outputs）的实现细节有差异，我们有一次切模型后发现 JSON 解析报错，就是参数兼容问题。做切换前建议先用测试用例跑一遍。
