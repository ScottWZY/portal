## 五、如何在项目中对比模型

### 5.1 建立评估集

1. 收集 20~50 个**真实用户提问**（从业务中采样）
2. 对每个提问整理**期望回答**或**评估标准**（正确/错误、相关/不相关）
3. 保存为 JSON 文件：

```json
[
  {
    "question": "如何申请年假？",
    "expected": "年假需要在OA系统申请，提前3天提交...",
    "relevant_documents": ["员工手册.pdf", "请假制度.md"]
  },
  ...
]
```

### 5.2 自动化对比流程

```python
# 伪代码：自动化对比不同模型
for model in ["deepseek-chat", "qwen-max", "gpt-4o"]:
    total = 0
    correct = 0
    for sample in eval_set:
        answer = call_model(model, sample["question"])
        if evaluate_answer(answer, sample["expected"]):
            correct += 1
        total += 1
    accuracy = correct / total
    cost = calculate_cost(model, eval_set)
    print(f"{model}: accuracy={accuracy:.2%}, cost=${cost:.2f}")
```

### 5.3 决策指标

- **准确率优先**：如果错了后果严重 → 选准确率最高的
- **成本优先**：如果错了可以人工纠正 → 选性价比最高的
- **延迟优先**：如果要求响应快 → 选小模型+缓存

## 面试高频题

### Q1: 如何科学地评估大模型的能力？常用的评测基准有哪些？各有什么局限性？
**详细答案：** 科学评估大模型能力需要多维度、多基准的综合考量，而非依赖单一指标。常用的评测基准分为几大类：(1) **知识理解类**：MMLU（57个学科的多选题，评估广泛知识覆盖）、CMMLU/C-Eval（中文版多学科理解）；(2) **代码能力类**：HumanEval（Python 函数补全，通过率衡量）、SWE-bench（真实 GitHub issue 修复，最接近实际编程能力）、MBPP（基础编程题）；(3) **数学推理类**：GSM8K（小学数学应用题）、MATH（竞赛数学题）；(4) **综合能力类**：MMMU（多模态多学科理解）、GPQA（研究生级别问答）。每个基准测试都从特定维度衡量模型能力，不应单独使用。

但所有评测基准都有局限性：(1) **数据污染**：许多基准的测试集数据可能已被包含在模型的训练数据中，导致"作弊"——模型在训练时见过答案，而非真正理解。这在高分模型中尤为常见。(2) **评测与实际脱节**：HumanEval 的 Python 函数补全与真实项目中的代码开发差距巨大，SWE-bench 虽然更接近实际，但只覆盖了 GitHub issue 修复这一特定场景。(3) **语言和文化偏见**：MMLU 等基准以英语和西方文化为中心，不能直接反映模型在中文环境和中文任务上的表现。(4) **静态评测**：基准测试是固定的，无法反映模型在新领域、新任务上的泛化能力。因此，外部基准只能作为参考，最终决策应基于项目自身的评估集（使用真实业务数据）。

建立项目评估集的方法：(1) 从业务中采样 20-50 个真实用户提问；(2) 人工标注期望回答或评估标准；(3) 使用 LLM-as-Judge 或人工评估各模型的回答质量；(4) 综合考量准确率、成本、延迟三个维度做出选择。这种方法虽然耗时，但远比依赖公开基准更可靠。

### Q2: 哪个模型的中文能力最好？为什么国产模型在中文评测上领先？
**详细答案：** 从 CMMLU 和 C-Eval 等中文评测基准来看，国产模型（Qwen3-Max、DeepSeek R1/V3.2）在中文能力上确实领先于国际模型（GPT-4o、Claude、Gemini）。Qwen3-Max 以 86.8% 的 CMMLU 得分位居第一，DeepSeek R1 和 V3.2 紧随其后，而 GPT-4o 和 Claude Opus 4.6 在 82-83% 左右。这一差距的原因主要有三点：(1) **训练数据比例**：国产模型在预训练阶段使用了更高比例的高质量中文语料，涵盖了中文互联网、学术文献、历史文化等领域的丰富文本，而国际模型的中文语料占比相对较低。(2) **中文特有知识**：中国的高中/大学学科知识体系（如中国历史、政治、文学）在国际模型的训练数据中覆盖不足，导致在 C-Eval 等考试类基准上表现不佳。(3) **中文语义理解**：中文的成语、典故、诗词、委婉表达等需要深厚的语言文化理解，国产模型在中文特有的语义理解和生成上更自然。

但这并不意味着所有中文任务都应选择国产模型。在实际项目中，选择取决于任务类型：(1) 如果是中文知识库问答、中文客服、中文内容生成等纯中文任务，国产模型（Qwen3-Max、DeepSeek V3.2）是更优选择；(2) 如果任务涉及代码生成、英文文档处理、多语言混合，Claude Sonnet 4.6 和 GPT-4.1 的表现可能更好；(3) 如果需要深度的推理和分析（如中文法律分析、中文医学诊断），建议使用 DeepSeek R1 或 o3 等推理模型。

一个实用的评估技巧：在项目中用中文和英文分别测试同一个模型，观察回答质量的差异。如果中文回答明显不如英文，说明该模型的中文语料和训练不足，应考虑换用国产模型或通过翻译层进行折中。

### Q3: Claude 在代码能力上为什么领先？SWE-bench 是什么？如何解读 SWE-bench 得分？
**详细答案：** Claude 在代码能力上的领先地位主要归功于 Anthropic 在代码训练和 Agent 能力上的持续投入。SWE-bench（Software Engineering Benchmark）是评估模型解决真实软件工程问题的基准测试，它从 GitHub 上收集了 2,294 个真实的 Python issue，要求模型在给定的代码仓库中定位 bug 并生成正确的代码修复补丁。SWE-bench 的通过率（resolved rate）衡量的是：模型生成的补丁是否通过了该 issue 对应的所有测试用例。这与 HumanEval 等简单代码补全测试有本质区别——SWE-bench 要求模型具备代码理解、上下文定位、多文件编辑、测试验证等综合能力，更接近真实的软件开发工作。

Claude Sonnet 4.6 以 72.7% 的 SWE-bench 通过率位居第一，Claude Opus 4.6 紧随其后（72.5%），远超 o3（55%）和 GPT-4.1（42%）。这一差距反映了 Anthropic 在 Agent 编程能力上的技术优势：Claude 的计算机使用能力（Computer Use）、工具调用、长上下文理解（1M tokens）和代码推理能力在 SWE-bench 场景下得到了充分发挥。Anthropic 还在训练中使用了大量代码相关的强化学习数据，优化了模型在"理解代码 → 定位问题 → 生成修复 → 验证正确性"这一完整链条上的表现。

解读 SWE-bench 得分需要注意：(1) 得分高不代表模型在所有编程任务上都更好，SWE-bench 偏向 bug 修复场景，不代表代码生成、架构设计、代码审查等其他能力；(2) 测试集是 Python 仓库，不能直接推广到其他语言；(3) 模型在 SWE-bench 上的表现与使用的 Agent 框架（如 SWE-Agent、Devin 等）和测试配置密切相关，不同的评估设置可能产生不同的得分。因此，SWE-bench 是代码能力的重要参考指标，但不应作为唯一标准。

### Q4: 在你的项目中如何对比和选择模型？说说完整的方法论和考量维度。
**详细答案：** 选择模型是一个多维度决策问题，需要综合考虑准确率、成本、延迟、上下文长度、安全性、生态兼容性等因素。完整的方法论分为五个步骤：(1) **明确需求**：列出项目的核心任务类型（对话、知识检索、代码生成、推理分析等），确定每个任务的优先级和评估标准。(2) **初筛模型**：基于公开基准和社区口碑，选出 3-5 个候选模型。例如，中文知识库问答候选为 Qwen3-Max、DeepSeek V3.2、GPT-4o；代码 Agent 候选为 Claude Sonnet 4.6、o3、DeepSeek R1。(3) **建立评估集**：从业务中采集 20-50 个真实场景的测试样例，人工标注期望回答的质量标准（正确/错误/部分正确），保存为结构化的评估数据集。(4) **自动化评估**：编写脚本对每个候选模型运行评估集，记录每个模型的准确率、响应时间、Token 消耗、API 费用。对于主观性较强的任务（如对话质量），使用 LLM-as-Judge 辅助评估。(5) **综合决策**：根据准确率、成本、延迟的权重做出最终选择。

考量维度及其权重取决于项目场景：(1) **准确率优先**（如医疗诊断、法律咨询）：错误后果严重，优先选准确率最高的模型，成本可适当放宽。(2) **成本优先**（如大规模客服、内容批量生成）：错误可人工纠正，优先选性价比高的模型（如 DeepSeek V3.2，价格仅为 GPT-4o 的 1/10）。(3) **延迟优先**（如实时对话、语音助手）：优先选小模型或快速推理方案（如 Gemini Flash、Llama 3B），配合缓存和流式输出。(4) **上下文长度**：处理超长文档（>100K tokens）时，优先选支持 1M 上下文的模型（Claude、Gemini、GPT-4.1）。(5) **数据安全**：数据不能出本地的场景，只能选可本地部署的模型（Qwen、DeepSeek、Llama 的开源版本）。

最终建议：不要追求"最好的模型"，而要追求"最适合你项目的模型"。最实用的方法是做一个简单的评估矩阵，给每个维度打分和加权，用数据驱动决策，而非凭感觉选择。

### Q5: 推理模型（如 o3、DeepSeek R1）和通用模型（如 GPT-4o、DeepSeek V3）的核心区别是什么？什么场景下需要推理模型？
**详细答案：** 推理模型和通用模型的核心区别在于思考方式。通用模型（如 GPT-4o、DeepSeek V3、Claude Sonnet）是"直接生成"模式——收到问题后立即开始生成回答，每一步预测下一个最可能的 token。推理模型（如 o3、DeepSeek R1）是"思考后生成"模式——在生成最终回答之前，会先进行一段内部推理过程（Chain of Thought，思维链），用额外的 token 进行分步推理、自我验证、多角度论证，然后基于推理过程输出最终答案。这种差异在技术实现上体现为：推理模型在训练时使用了大量的强化学习（RL）和思维链数据，学会了"慢思考"的能力。

推理模型的核心优势在于处理需要多步推理、逻辑分析、数学证明等复杂任务。在 GSM8K 和 MATH 基准上，o3（96.3%/78.5%）和 DeepSeek R1（95.1%/74.8%）的得分显著高于通用模型。具体适用场景包括：(1) **数学和科学问题**：需要多步推导、公式计算、逻辑证明的题目；(2) **复杂代码调试**：需要逐步分析代码逻辑、定位 bug 根因、验证修复方案的任务；(3) **法律和医学分析**：需要综合多源信息、严格逻辑推理的专业判断；(4) **多条件决策**：涉及多个约束条件和权衡的复杂决策场景。

但推理模型也有代价：(1) **成本更高**：推理过程中产生的额外 token（思维链）需要付费，单次调用的 token 消耗可能是通用模型的 3-10 倍；(2) **延迟更高**：推理过程需要更多时间，首 token 延迟显著增加；(3) **简单任务浪费**：对于"你好"、"今天天气怎么样"等简单问题，推理模型也会进行不必要的思考，造成浪费。因此，最佳实践是：对简单任务使用通用模型（快速、便宜），对复杂推理任务使用推理模型（准确、可靠），或通过路由机制（根据问题复杂度自动选择模型）实现动态切换。在实际项目中，DeepSeek R1 以极低的价格（约为 o3 的 1/20）提供了接近 o3 的推理能力，是目前性价比最高的推理模型选择。
