# 分布式 ID

> 内容完善中，此页面将持续更新。

## 概述

在分布式系统中，经常需要生成全局唯一的 ID 来标识数据（如订单号、用户 ID、消息 ID 等）。由于分布式系统没有单点数据库的自增主键，需要专门的方案来生成全局唯一且有序的 ID。

## 核心方案概览

### UUID

UUID（Universally Unique Identifier）是一种标准化的全局唯一标识符，通过随机数或机器信息生成 128 位的 ID。

**优点**：
- 生成简单，无需任何中心化协调
- 冲突概率极低（理论上可忽略）

**缺点**：
- 无序，不适合作为数据库主键（B+树索引分裂严重）
- 占用空间大（36 字符 / 128 位）
- 不可读，不利于问题排查

### 雪花算法（Snowflake）

Twitter 开源的分布式 ID 生成算法，产生 64 位 Long 类型 ID，结构如下：

```
┌─┬─────────────────────────────────────────────────────────────────────────┐
│0│ 41 位时间戳 (毫秒)  │ 10 位机器 ID  │ 12 位序列号 │
└─┴─────────────────────────────────────────────────────────────────────────┘
```

| 组成部分 | 位数 | 说明 |
|----------|------|------|
| **符号位** | 1 位 | 始终为 0，表示正数 |
| **时间戳** | 41 位 | 毫秒级时间戳，可使用约 69 年 |
| **机器 ID** | 10 位 | 可部署 1024 台机器 |
| **序列号** | 12 位 | 同一毫秒内可生成 4096 个 ID |

**优点**：
- 高性能（纯内存计算，单机可达数万 QPS）
- 趋势递增，适合数据库索引
- 可定制化（调整各段位数）

**缺点**：
- 依赖机器时钟，时钟回拨会导致 ID 重复
- 机器 ID 需要手动分配或额外协调

### 数据库号段模式

利用数据库的自增能力或号段分配来生成 ID。常见实现：每次从数据库申请一批 ID 号段（如 1000 个），在本地内存中分配使用，用完后再次申请。

**优点**：
- 强依赖数据库，ID 分配可靠
- 趋势递增，适合数据库索引

**缺点**：
- 数据库成为单点瓶颈
- 号段用尽时需重新申请，存在延迟

### Redis 生成

利用 Redis 的 `INCR` 或 `INCRBY` 命令原子递增生成 ID。

**优点**：
- 高性能（Redis 单机可达 10 万+ QPS）
- 天然递增

**缺点**：
- 依赖 Redis 的高可用（需持久化 + 主从）
- 恢复后可能重复（需设置合适的初始值）

## 方案对比

| 方案 | 性能 | 有序性 | 依赖性 | 复杂度 |
|------|------|--------|--------|--------|
| **UUID** | 极高 | 无序 | 无 | 低 |
| **雪花算法** | 高 | 趋势递增 | 依赖时钟 | 中 |
| **数据库号段** | 中 | 严格递增 | 依赖数据库 | 中 |
| **Redis** | 高 | 严格递增 | 依赖 Redis | 中 |

## 选型考量

选择分布式 ID 方案时，需要综合考虑以下因素：

- **是否需要有序**：数据库主键通常需要有序 ID 以优化 B+树索引
- **性能要求**：ID 生成的 QPS 需求
- **可用性要求**：ID 生成服务是否允许短暂不可用
- **运维复杂度**：是否需要额外的中间件支持
