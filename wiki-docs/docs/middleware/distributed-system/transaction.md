# 分布式事务

> 内容完善中，此页面将持续更新。

## 概述

分布式事务是指在分布式系统中，一个业务操作需要跨越多个独立的服务或数据库，且需要保证这些操作要么全部成功，要么全部失败，以维持数据一致性。

## 核心方案概览

### 2PC（两阶段提交）

两阶段提交（Two-Phase Commit）是实现分布式事务的经典协议，通过协调者（Coordinator）统一管理参与者（Participants）的提交或回滚。

- **阶段一（Prepare）**：协调者向所有参与者询问是否可以提交事务。
- **阶段二（Commit/Abort）**：协调者根据投票结果决定提交或回滚。

::: warning 核心缺陷
协调者单点故障时，已投 Yes 的参与者会被永久阻塞，无法知道最终是 Commit 还是 Abort。
:::

### 3PC（三阶段提交）

在 2PC 的基础上增加了**预提交（PreCommit）**阶段，并引入超时机制，部分缓解了协调者单点故障导致的阻塞问题。

### TCC（Try-Confirm-Cancel）

TCC 是一种柔性事务方案，将事务分为三个阶段：

| 阶段 | 说明 |
|------|------|
| **Try** | 资源预留和锁定，检查业务可行性 |
| **Confirm** | 确认执行业务操作，使用 Try 阶段预留的资源 |
| **Cancel** | 取消业务操作，释放 Try 阶段预留的资源 |

TCC 适用于需要强隔离性的业务场景，如金融转账、库存扣减等。

### Saga 模式

Saga 将长事务拆分为多个本地事务，每个本地事务都有对应的补偿操作。如果某个本地事务失败，则按逆序执行之前成功事务的补偿操作。

**两种协调模式**：

| 模式 | 说明 | 优缺点 |
|------|------|--------|
| **编排（Choreography）** | 各服务通过事件驱动自主协调 | 松耦合，但流程分散难以追踪 |
| **控制（Orchestration）** | 由 Saga 编排器统一调度 | 集中可控，但编排器成为单点 |

### Seata

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里巴巴开源的分布式事务解决方案，提供 AT、TCC、Saga 和 XA 四种事务模式。

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| **AT** | 自动回滚，对业务无侵入 | 基于关系型数据库的微服务 |
| **TCC** | 手动定义 Try/Confirm/Cancel | 对隔离性要求高的场景 |
| **Saga** | 长事务补偿 | 多服务长流程 |
| **XA** | 基于 XA 协议 | 需要强一致性的场景 |

## 选型考量

选择分布式事务方案时，需要综合考虑以下因素：

- **一致性要求**：强一致性还是最终一致性？
- **性能要求**：事务协调对吞吐量和延迟的影响？
- **业务侵入性**：是否需要大量修改业务代码？
- **运维复杂度**：分布式事务组件的部署和维护成本？
