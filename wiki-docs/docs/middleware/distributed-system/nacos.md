# Nacos 核心

> **最新版本**：Nacos 3.2.x（2026年），AI原生 + AP/CP双模增强。

> **概述**：Nacos（Dynamic Naming and Configuration Service）是阿里巴巴开源的动态服务发现、配置管理和服务管理平台。Nacos 的名字来源于 **Na**ming + **Co**nfiguration + **S**ervice。

## 知识图谱

### 入门级：基础概念

#### Nacos 是什么

Nacos 是阿里巴巴在 2018 年开源的云原生中间件，定位为**动态服务发现、配置管理和服务管理平台**。它集合了 Eureka（服务注册发现）、Consul（服务健康检查）和 Apollo（配置管理）的核心能力，以"一个平台解决多个问题"的理念降低微服务架构中的组件碎片化问题。

Nacos 的核心设计哲学：

- **服务是核心**：围绕"服务"这一微服务架构的一等公民，提供全生命周期管理。
- **动态是关键**：支持实时感知服务状态变化，实现配置的动态刷新而不重启应用。
- **云原生优先**：从架构设计之初就考虑了容器化、Kubernetes 等云原生场景的适配。

Nacos 在 2023 年毕业成为 CNCF 顶级项目，截至 2026 年已在全球超过 10 万家企业中部署使用。

#### 基础概念表

| 概念 | 英文 | 说明 |
|------|------|------|
| **命名空间** | Namespace | 用于实现多租户隔离，不同 Namespace 下的数据完全隔离 |
| **分组** | Group | 服务和配置的逻辑分组 |
| **服务** | Service | 一组具备相同功能的实例集合 |
| **实例** | Instance | 服务中的一个具体节点，具有 IP、端口、权重、健康状态等属性 |
| **配置** | Configuration | 应用运行时的动态可变更参数 |
| **集群** | Cluster | 一组 Nacos 节点的集合，分为 AP 集群和 CP 集群 |
| **健康检查** | Health Check | 对实例可用性的检测机制 |
| **元数据** | Metadata | 服务和实例的扩展信息，可用于灰度路由、流量染色 |

#### 典型应用场景

1. **微服务注册发现**：Spring Cloud Alibaba 生态中作为默认注册中心
2. **动态配置管理**：实现配置集中管理和实时刷新
3. **灰度发布与流量管理**：结合元数据和路由规则，实现按标签的流量调度
4. **DNS-Based 服务发现**：支持将服务名解析为 IP 列表
5. **多环境隔离**：通过 Namespace 实现开发/测试/预发/生产环境隔离

### 进阶级：原理机制

#### 服务注册流程

```
  ┌──────────┐     POST /nacos/v1/ns/instance    ┌──────────────┐
  │  客户端    │ ────────────────────────────────> │  Nacos Server  │
  │ (Provider) │ <──────────────────────────────── │  (注册中心)     │
  └──────────┘         返回成功 + 实例信息           └──────────────┘
       │                                                   │
       │ ① 构造 Instance 对象（IP/Port/Weight/Meta）         │
       │ ② 发送注册请求                                     │
       │ ③ Server 收到请求                                  │
       │                                                   │ ④ 写入内存注册表
       │                                                   │ ⑤ 临时实例 → Distro(AP)
       │                                                   │ ⑥ 持久实例 → Raft(CP)
       │                                                   │ ⑦ 返回注册成功
       │                                                   │ ⑧ 推送变更给订阅者
```

#### 服务发现流程

Nacos 支持两种模式：
- **UDP 推送**（本机）：服务端变更时通过 UDP 广播通知同机器上的客户端
- **HTTP 轮询**：客户端定时（默认 10s）向服务端拉取最新的服务实例列表

#### 配置管理流程（长轮询机制）

Nacos 配置中心的动态刷新是基于 **长轮询（Long Polling）** 实现的：

```
  客户端                      Nacos 服务端
  ────────                    ────────────
    │ ① GET /nacos/v1/cs/configs/listener
    │    (携带 MD5 值列表)          │
    │ ───────────────────────────> │
    │                             │ ② 比对 MD5，有变更立即返回
    │                             │ ③ 无变更则挂起 30s (Long Polling)
    │    (30s 内若有变更即时返回)    │
    │ <───────────────────────────│ ④ 返回变更的 dataId+group
    │ ⑤ 拉取变更后的配置内容         │
    │ ⑥ 触发 @RefreshScope 刷新     │
```

长轮询的优势：低延迟、低开销、兼容性好。

### 经典高频级：核心难点

- **Distro 协议深入理解**：最终一致性原理、版本号对账、健康检查去重
- **AP/CP 双模式**：临时实例 vs 持久实例的选型与权衡
- **配置动态刷新**：长轮询 → @RefreshScope → 事件驱动三层协同
- **健康检查机制**：临时实例心跳 vs 持久实例服务端探测
- **Nacos 3.x 新特性**：AI 原生、百万级实例、K8s 原生支持

## 核心原理深度解析

### 1. Distro 协议详解

Distro 协议是 Nacos 自研的**最终一致性分布式协议**，专门为服务发现的 AP 场景设计。

```
                    ┌─────────────────────────────────────┐
                    │         Distro 协议架构              │
                    └─────────────────────────────────────┘

      Node-A ◄──────── Peer-to-Peer (对等架构) ────────► Node-B
        │                                                      │
        ├── 接收写请求     (任意节点可写)                        ├── 接收写请求
        ├── 同步到其他节点 (异步 + 重试)                        ├── 同步到其他节点
        ├── 版本号对账     (定期全量/增量)                       ├── 版本号对账
        └── 健康检查去重   (按节点分摊)                          └── 健康检查去重
```

**五大核心机制**：

1. **对等无 Leader 架构**：所有节点地位对等，任意节点都可接收客户端的写请求。对比 Raft 协议：Raft 所有写操作必须经过 Leader，Leader 宕机时需要选举，期间无法写入。

2. **写操作流程**：
   ```
   客户端 → Node-X → 写入本地数据存储 → 立即返回成功（不等同步结果）
                     → 异步发送 DistroTask 给其余节点
                     → 目标节点写入本地并返回 ACK
                     → 若同步失败，写入重试队列，延迟重试
   ```
   关键点：**先写本地，异步同步**。这意味着写操作延迟约等于单机写入延迟。

3. **异步同步机制**：同步任务以 `DistroTask` 形式放入任务队列，默认重试 3 次，每次间隔递增。

4. **版本号对账机制**：每个服务实例数据携带一个单调递增的版本号。节点间定期（默认 5s）进行全量对账 + 增量对账。

5. **健康检查去重**：Nacos 按 `hash(serviceName) % clusterSize` 将健康检查任务分摊到不同节点。

#### Distro vs Raft 对比

| 对比维度 | Distro | Raft |
|---------|--------|------|
| **一致性级别** | 最终一致 | 强一致 |
| **写入方式** | 任意节点可写，异步同步 | 必须经过 Leader，同步提交 |
| **写入延迟** | 低（本地写即返回） | 较高（需多数节点确认） |
| **Leader 依赖** | 无 Leader，对等架构 | 依赖 Leader，选举时不可用 |
| **数据对账** | 版本号驱动的定期对账 | Raft Log 天然保证 |
| **适用场景** | 服务注册发现 | 配置管理 |

#### 为什么 Distro 没有采用 Raft？

1. **写入必须经过 Leader**：如果 Leader 所在节点网络故障，在选举完成前整个集群都无法写入。
2. **选举延迟**：Leader 选举通常需要 150ms~300ms，在此期间服务注册请求会失败。
3. **跨地域部署困难**：Raft 要求多数节点确认写入，跨地域部署时延迟会显著增加。
4. **复杂度高**：Raft 的状态机、日志压缩、快照等机制增加了运维难度。

### 2. AP/CP 双模式切换

Nacos 是业界少数同时支持 AP 和 CP 两种模式的注册中心。

#### 临时实例 vs 持久实例

```
          ┌──────────────────────────────────────────────────────┐
          │              Nacos 实例类型决策树                      │
          └──────────────────────────────────────────────────────┘
                                │
              ┌─────────────────┴─────────────────┐
              ▼                                   ▼
      ┌───────────────┐                   ┌───────────────┐
      │   临时实例      │                   │   持久实例      │
      │ (ephemeral)   │                   │ (persistent)   │
      ├───────────────┤                   ├───────────────┤
      │ 默认类型       │                   │ 需显式指定      │
      │ AP (Distro)   │                   │ CP (Raft)      │
      │ 客户端心跳      │                   │ 服务端主动探测   │
      │ 不可用则自动剔除 │                   │ 不可用只标记不删除 │
      │ 适合微服务节点   │                   │ 适合 DNS 解析等  │
      └───────────────┘                   └───────────────┘
```

#### 对比表

| 特性 | 临时实例（AP） | 持久实例（CP） |
|------|---------------|---------------|
| **一致性协议** | Distro（自研AP协议） | Raft（强一致性） |
| **心跳机制** | 客户端主动上报（5s间隔） | Nacos服务端主动探测 |
| **心跳超时** | 15s 无心跳标记不健康 → 30s 后自动剔除 | 探测失败仅标记不健康，不自动删除 |
| **数据一致性** | 最终一致性（秒级） | 强一致性（写入即一致） |
| **集群耐受** | 容忍多数节点故障 | 必须保持多数派存活 |
| **适用场景** | 微服务节点（动态扩缩） | DNS-F模式、核心网关路由 |

**选型建议**：

```
是否需要强一致性？
     ├── 是 → 持久实例 (CP)
     │       场景：DNS 解析、核心流量入口、金融交易
     │
     └── 否 → 是否允许自动剔除？
              ├── 是 → 临时实例 (AP) [推荐]
              │       场景：微服务节点、弹性伸缩、Serverless
              │
              └── 否 → 持久实例 (CP)
```

### 3. 配置动态刷新原理

Nacos 的配置动态刷新是一个**三层协同机制**：

```
    ┌─────────────────────────────────────────────────────────────┐
    │                 Nacos 配置动态刷新三层模型                     │
    └─────────────────────────────────────────────────────────────┘

    第一层：长轮询 (Long Polling)           ← 传输通道
    客户端发起 HTTP 请求 → 服务端挂起 30s → 有变更返回 / 超时重来

    第二层：@RefreshScope                 ← 对象生命周期管理
    Spring Cloud 提供的注解，标记的 Bean 在配置变更时会被重新创建
    原理：CGLIB 代理 → 缓存失效 → 下次访问时重建 Bean

    第三层：事件驱动 (RefreshEvent)         ← 自定义扩展
    配置变更后发布 RefreshScopeRefreshedEvent 事件
    应用可通过 @EventListener 监听并执行自定义逻辑
```

#### 第一层：长轮询机制详解

```
         客户端                                   Nacos 服务端
         ──────                                   ────────────
    ① 发起监听请求
    POST /nacos/v1/cs/configs/listener
    { "Listening-Configs": "dataId^2group^2MD5值\n...",
      "Long-Pulling-Timeout": "30000" }
    ─────────────────────────────────────────────►
                                                   ② 服务端收到请求
                                                   ③ 比对 MD5，判断是否有变更
                                                   ④ 若有变更 → 立即返回
                                                   ⑤ 若无变更 → 挂起 30s
    ◄─────────────────────────────────────────────
    ⑥ 收到响应（变更列表 或 空表示超时）
         │
         ├── 有变更 → GET /nacos/v1/cs/configs 拉取新配置
         │            → 触发 RefreshScope 刷新
         │
         └── 无变更 → 重新发起监听请求（回到 ①）
```

**长轮询 vs 短轮询 vs WebSocket**：

| 方案 | 实时性 | 资源消耗 | 实现复杂度 | 防火墙友好 |
|------|--------|---------|-----------|-----------|
| 短轮询 | 差 | 高 | 低 | 是 |
| 长轮询 | 高（秒级） | 低 | 中 | 是 |
| WebSocket | 最高 | 最低 | 高 | 部分场景受限 |

Nacos 选择长轮询的核心原因：兼容性好、资源效率高、实现简单。

#### 第二层：@RefreshScope 原理

1. **代理包装**：被注解的 Bean 会被 CGLIB 生成代理对象。
2. **懒加载缓存**：代理对象内部维护一个 `ObjectFactory`，首次访问时创建真实 Bean 并缓存。
3. **缓存失效**：监听到 `RefreshEvent` 时，调用 `ScopeCache.clear()` 清空缓存。
4. **延迟重建**：下一次用户请求访问该 Bean 时，代理重新创建新 Bean，注入最新配置值。

::: warning 注意
`@RefreshScope` 仅对 `@Value` 注入的属性生效。`@ConfigurationProperties` 绑定的类**无需** `@RefreshScope`，Spring Boot 内置机制会自动处理。
:::

### 4. 健康检查机制

#### 临时实例：客户端心跳机制

```
        临时实例                            Nacos Server
       (客户端)                             (服务端)
    ┌──────────┐                        ┌──────────────┐
    │ 心跳线程   │ ───[心跳包 每5s]───> │   健康检查模块  │
    │          │                        │              │
    │          │                        │ ① 记录心跳时间 │
    │          │                        │ ② 更新 lastBeat│
    └──────────┘                        └──────┬───────┘
                                    ┌───────────▼──────────┐
                                    │  定期健康检查(每5s)    │
                                    │  now - lastBeat       │
                                    │  < 15s → 健康          │
                                    │  15s~30s → 不健康(保留) │
                                    │  > 30s → 自动剔除       │
                                    └──────────────────────┘
```

**关键时间参数**：心跳间隔 5s，健康超时 15s，剔除超时 30s。

#### 持久实例：服务端探测机制

支持三种探测类型：TCP 连接探测、HTTP GET 探测、MySQL 连接探测。探测失败仅标记不健康，不会自动删除。

| Kubernetes 环境 | Nacos（K8s 原生支持更好） |

## 经典高频面试题

### Q1：Nacos 的 AP 模式和 CP 模式有什么区别？如何选择？

**答案**：

| 对比维度 | AP 模式 | CP 模式 |
|---------|--------|--------|
| **一致性协议** | 自研 Distro 协议（最终一致） | Raft 协议（强一致） |
| **实例类型** | 临时实例（ephemeral=true，默认） | 持久实例（ephemeral=false） |
| **Leader** | 无 Leader，对等架构，任意节点可读写 | 有 Leader，写操作必须经 Leader 提交 |
| **写入一致性** | 最终一致性（秒级收敛） | 强一致性（写入即一致） |
| **网络分区表现** | 分区两端都可独立提供服务 | 少数派分区不能写入、不能选主 |
| **健康检查** | 客户端心跳上报，超时自动删除 | 服务端主动探测，仅标记状态 |

**从请求路径理解两种模式**：

AP 模式（Distro）的写操作路径：
```
客户端 → 任意 Nacos 节点 → 本地写入 → 立即返回成功 → 异步同步到其他节点
```
核心思想：**先保证可用性，后追求一致性**。

CP 模式（Raft）的写操作路径：
```
客户端 → Nacos Leader 节点 → 同步到 Follower → 多数节点确认 → 返回成功
```
核心思想：**先保证一致性，后考虑可用性**。

**选型指南**：

1. **优先选择 AP 模式**：微服务节点弹性扩缩容、Serverless、网络环境不稳定场景
2. **选择 CP 模式**：DNS-F 模式服务解析、核心网关路由表、金融级服务治理
3. **混合使用**：同一集群中部分服务用临时实例（AP），部分用持久实例（CP）

### Q2：Nacos 配置中心如何实现配置动态刷新？

**答案**：

Nacos 配置动态刷新的核心是一个**三层协同机制**：长轮询（传输通道）→ @RefreshScope（Bean 生命周期）→ 事件驱动（自定义扩展）。

**第一层：长轮询（Long Polling）**：客户端发起 HTTP 请求携带当前配置的 MD5 列表，服务端比对 MD5，有变更立即返回，无变更则挂起 30s。使用 Servlet 3.0 AsyncContext 挂起请求而不阻塞线程。

**第二层：@RefreshScope**：CGLIB 代理包装 Bean → 懒加载缓存 → 收到 RefreshEvent 时清空缓存 → 下次访问时重建 Bean 并注入最新配置值。

**第三层：事件驱动**：

| 事件 | 时机 | 用途 |
|------|------|------|
| `RefreshEvent` | Bean 被标记为 dirty | 触发 RefreshScope 清除缓存 |
| `RefreshScopeRefreshedEvent` | 所有 RefreshScope Bean 刷新完成 | 重建连接池、清空本地缓存 |
| `EnvironmentChangeEvent` | Environment 层面属性变更 | 监控配置项变更并告警 |

### Q3：Nacos 如何防止服务注册中的假死问题？

**答案**：

**Nacos 的防假死机制**：

**1. 临时实例：分阶段心跳处理，避免误判**

```
  客户端心跳发送 (5s) → 服务端状态判断:
         
  上次心跳距现在 < 15s            → HEALTHY (健康)
  上次心跳距现在 15s ~ 30s        → UNHEALTHY (不健康, 但保留)
  上次心跳距现在 > 30s            → 剔除实例
```

Nacos 引入了 **15s 的"不健康但不剔除"缓冲区**：
- 即使连续 3 次心跳丢失（最多 15s），实例也不会被立即删除。
- 只有持续 30s 无心跳才会剔除。
- 在"不健康"状态期间，如果心跳恢复，实例会立即变回 HEALTHY。

**2. 持久实例：仅标记状态，从不自动删除**

**Nacos vs Eureka 假死处理对比**：

| 维度 | Nacos | Eureka |
|------|-------|--------|
| **核心机制** | 分层超时（不健康缓冲区 15s + 剔除 30s） | 自我保护模式 + 心跳超时 |
| **自我保护** | 无专门的自我保护模式 | 核心特色功能 |
| **心跳超时** | 30s 无心跳剔除 | 默认 90s 无心跳剔除 |
| **误剔除风险** | 低（缓冲区 + 用户可配置） | 中（自我保护模式正确配置下较低） |

**为什么 Nacos 没有自我保护模式？**
- Nacos 的临时实例已经通过 15s 缓冲区 + 30s 剔除的超时设计，在网络短暂抖动时不会误剔除。
- Nacos 的 AP 模式采用 Distro 协议，网络分区两端可独立工作，分区恢复后通过版本号对账自动合并数据。
- Nacos 认为避免大规模误剔除的责任应该在机制设计层面解决，而非引入一个"关闭剔除"的开关。

### Q4：Distro 协议是如何工作的？

**答案**：

Distro 是 Nacos 自研的**最终一致性 AP 协议**，核心思想：**对等无 Leader 架构 + 异步同步 + 版本号对账**。

**完整写操作流程**：

```
步骤1: 客户端向 Node-A 发起服务注册请求
步骤2: Node-A 执行本地写入（更新内存注册表 + 更新版本号）
步骤3: Node-A 立即返回成功给客户端（不等待同步结果）
步骤4: 异步同步阶段
         ├── Node-A 构造 DistroTask
         ├── 放入同步队列 → 向 Node-B 发送 HTTP 请求
         ├── Node-B 收到后检查版本号 → 更新本地注册表
         ├── 同步失败 → 进入重试队列 (默认 3 次)
         └── 重试仍失败 → 进入 Dead Queue
```

**版本号对账机制（核心可靠性保障）**：

```
① 全量对账（每 5s）
   Node-A 随机选择对账目标 Node-B → 请求版本号摘要 → 比对差异 → 增量同步

② 增量对账
   仅对版本号有差异的服务进行数据同步，避免全量数据传输
```

**健康检查去重**：负责节点 = hash(serviceName) % clusterSize，避免"检查风暴"。

### Q5：Nacos 和 Eureka 的区别是什么？

**答案**：

| 维度 | Nacos | Eureka |
|------|-------|--------|
| **开发厂商** | 阿里巴巴 | Netflix |
| **开源状态** | 活跃维护（3.x 2026年） | 2.0 停止开发（2018年停更） |
| **核心功能** | 服务发现 + 配置中心 + 服务治理 | 仅服务注册发现 |
| **CAP 模型** | AP/CP 双模式可切换 | 仅 AP |
| **一致性协议** | Distro(AP) + Raft(CP) | Peer to Peer(AP) |
| **健康检查** | 客户端心跳(临时) + 服务端探测(持久) | 客户端心跳 |
| **配置管理** | 原生内置、功能完善 | 无 |
| **实例类型** | 临时实例 + 持久实例 | 只有一种实例类型 |
| **Kubernetes** | 3.x 原生 Operator + xDS 协议 | 需 Sidecar 适配 |
| **社区活跃度** | 高（CNCF 项目） | 低（停止维护） |

**核心差异**：

1. **功能定位**：Eureka 是纯服务注册发现，Nacos 是"三合一"（服务发现 + 配置管理 + 服务治理）。
2. **一致性模型**：Eureka 纯 AP，Nacos AP/CP 可切换。
3. **健康检查**：Eureka 只支持客户端心跳，Nacos 支持心跳 + 服务端探测。

**迁移建议**：新项目首选 Nacos。已有的 Eureka 项目可以平滑迁移到 Nacos（Nacos 提供 Eureka 兼容 API）。

### Q6：Nacos 3.x 相比 2.x 有哪些核心升级？

**答案**：

Nacos 3.x（3.0 于 2025 年发布，3.2.x 于 2026 年发布）是一次里程碑式的架构升级，核心升级方向聚焦在五个方面：

**1. AI 原生（AI Native）**：智能路由、异常检测、容量预测、自然语言交互、智能配置推荐、根因分析。

**2. 性能大幅提升**：

| 指标 | Nacos 2.x | Nacos 3.x | 提升倍数 |
|------|-----------|-----------|---------|
| 最大实例数 | ~10万 | ~100万+ | 10x+ |
| 注册 TPS | ~5,000 | ~50,000 | 10x |
| 配置推送延迟 | 1~3s | <500ms | 3~6x |
| 内存占用(万实例) | ~2GB | ~500MB | 4x |

**3. Kubernetes 原生**：官方 Operator、xDS 协议支持、ConfigMap 双向同步、Sidecar 自动注入。

**4. 安全增强**：mTLS 全面支持、命名空间级 RBAC、完整审计链路、AES-256 加密。

**5. 可观测性**：OpenTelemetry Traces、Prometheus Metrics、Grafana Dashboard、集群健康度评分。

**升级建议**：Nacos 2.x 可以直接升级到 3.x（兼容 2.x 的 API 和数据格式）。建议先在非核心环境验证 2 周。风险点：gRPC 长连接模型变化可能导致现有网络策略需要调整。
