# 消息队列概览

> 消息队列是分布式系统中实现异步通信、系统解耦和流量削峰的核心中间件。

---

## 一、为什么需要消息队列

在现代分布式架构中，消息队列（Message Queue，简称 MQ）解决了三大核心问题：

### 1.1 异步解耦

在没有消息队列的系统中，服务间直接通过 RPC 或 HTTP 同步调用，形成强耦合的调用链。一旦某个下游服务不可用或响应变慢，整个调用链都会被阻塞。

```
同步调用链（无 MQ）：
  Service A ──RPC──> Service B ──RPC──> Service C ──RPC──> Service D
       ↑                                                       |
       └──────────────── 任一环节阻塞，全线等待 ────────────────┘

异步解耦（有 MQ）：
  Service A ──发送消息──> [Message Queue]
                              |
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
     Service B           Service C           Service D
     (独立消费)           (独立消费)           (独立消费)
```

- **耦合度**：从"代码级依赖"降级为"数据级依赖"，各服务只需关心消息格式，无需感知对方的存在。
- **可用性**：下游服务宕机不影响上游服务的正常运行，消息在队列中暂存，下游恢复后继续消费。
- **可扩展性**：新增消费者无需修改生产者代码，只需订阅对应的 Topic 即可。

### 1.2 削峰填谷

在高并发场景（如秒杀、大促）中，瞬时流量可能是平时流量的数十倍甚至上百倍。如果后端服务直接面对这些流量冲击，会导致系统崩溃。

```
流量曲线：

  QPS
  ▲
  │     ╱╲
  │    ╱  ╲          ← 瞬时高峰（如秒杀开始瞬间）
  │   ╱    ╲
  │  ╱      ╲
  │ ╱        ╲___───
  │╱                ╲___───
  └──────────────────────────> 时间

  无 MQ：后端直接承受峰值 → 系统崩溃
  有 MQ：消息在队列中暂存 → 后端按自身能力匀速消费
```

**消息队列充当"蓄水池"**：上游流量涌入队列，下游按自身处理能力匀速消费，保护后端系统不被冲垮。

### 1.3 消息模型

消息队列的核心消息模型有两种：

| 模型 | 说明 | 类比 | 典型实现 |
|------|------|------|----------|
| **点对点（P2P / Queue）** | 一条消息只能被一个消费者消费一次，消费后消息从队列中移除 | 快递签收 | Kafka Consumer Group、RocketMQ 集群消费 |
| **发布-订阅（Pub/Sub）** | 一条消息可以被多个订阅者同时消费，每个订阅者独立接收完整消息 | 报纸订阅 | Kafka 多 Consumer Group、RocketMQ 广播消费 |

## 二、Kafka vs RocketMQ vs RabbitMQ 对比选型

### 2.1 核心对比表

| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|----------|
| **开发语言** | Java / Scala | Java | Erlang |
| **定位** | 分布式流处理平台 | 金融级业务消息中间件 | 通用消息代理 |
| **单机吞吐量** | 百万级 TPS | 十万级 TPS | 万级 TPS |
| **延迟** | 毫秒级 | 毫秒级 | 亚毫秒级 |
| **消息可靠性** | 高（多副本 + ISR） | 极高（同步刷盘 + 同步双写） | 高（镜像队列） |
| **消息顺序** | 分区内有序 | 分区内有序（全局有序可选） | 队列内有序 |
| **事务消息** | 支持（事务 API，0.11+） | 原生支持（半消息 + 回查） | 需插件支持 |
| **延迟消息** | 需插件自行实现 | 原生支持（18级 + 5.x 任意精度） | 原生支持（死信队列 + TTL） |
| **流处理** | 原生支持（Kafka Streams / KSQL） | 需配合 Flink 等外部组件 | 不支持 |
| **批量消息** | 原生支持 | 支持（有体积限制 ~4MB） | 支持 |
| **管理控制台** | 第三方工具（CMAK 等） | 内置 Web Console | 内置 Web Console |
| **社区生态** | 极其活跃，Apache 顶级项目 | 活跃，Apache 顶级项目 | 活跃，Pivotal 维护 |
| **部署复杂度** | 中等（4.0 后无需 ZooKeeper） | 中等（NameServer 极简） | 较低 |
| **适用场景** | 大数据管道、日志收集、流计算、CDC | 在线业务消息、分布式事务、金融场景 | 低延迟、复杂路由、企业集成 |

### 2.2 选型决策指南

```
                    需要消息队列？
                          │
            ┌─────────────┼─────────────┐
            ▼             ▼             ▼
     大数据/日志/流计算   在线业务消息    低延迟/复杂路由
            │             │             │
            ▼             ▼             ▼
         Kafka         RocketMQ      RabbitMQ
      (百万级吞吐)    (事务+顺序)    (亚毫秒延迟)
            │             │             │
     日志采集 ELK    电商下单支付    企业集成
     CDC 数据同步    分布式事务       RPC 回调
     实时流计算      数据同步         IoT 消息
     事件溯源        削峰填谷         AMQP 标准
```

### 2.3 存储设计对比

| 维度 | Kafka | RocketMQ |
|------|-------|----------|
| **存储模型** | Partition 级分段日志（Segment） | 全局 CommitLog + ConsumeQueue 索引 |
| **写入方式** | 每个 Partition 独立顺序追加 | 全局顺序追加（所有 Topic 共享） |
| **文件数量** | 多（Partition 数 x Segment 数） | 少（仅 CommitLog 大文件 + 轻量索引） |
| **海量 Topic** | 不友好（文件数爆炸） | 友好（Topic 创建成本极低） |
| **零拷贝** | sendfile 系统调用 | mmap + FileRegion |
