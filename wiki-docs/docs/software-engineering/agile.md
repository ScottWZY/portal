## 一、敏捷开发的核心思想

### 1.1 敏捷不是"快"

敏捷（Agile）常被误解为"快速交付"。敏捷的真正核心是**快速响应变化**——接受需求会变化这一事实，建立能够灵活应对变化的开发节奏和组织结构。

::: tip 面试金句
"敏捷不是让你跑得更快，而是让你在方向错误时能更早地发现并转向。敏捷的核心是**反馈循环**（Feedback Loop）——越早获得反馈，纠偏成本越低。"
:::

### 1.2 敏捷宣言

> **个体和互动** 高于 流程和工具
> **可工作的软件** 高于 详尽的文档
> **客户合作** 高于 合同谈判
> **响应变化** 高于 遵循计划

虽然右项也有价值，但更重视左项的价值。

### 1.3 敏捷 vs 瀑布

| 维度 | 瀑布模型 | 敏捷开发 |
|------|---------|---------|
| **需求** | 一次性确定，变更成本高 | 迭代演化，欢迎变化 |
| **交付** | 项目结束时一次性交付 | 每个 Sprint 交付可工作的增量 |
| **反馈** | 项目尾声才获得用户反馈 | 每个 Sprint 获取反馈并调整 |
| **风险** | 集中在后期暴露 | 持续暴露，早期化解 |
| **适用场景** | 需求明确且稳定（航天、医疗设备） | 需求不确定或快速变化（互联网产品） |

## 二、Scrum 框架

Scrum 是最流行的敏捷框架，定义了角色、事件和工件三类核心要素。

### 2.1 三大角色

| 角色 | 职责 | 常见误解 |
|------|------|----------|
| **Product Owner（PO）** | 定义产品愿景、管理 Backlog 优先级、回答团队的业务疑问 | PO 不是"需求翻译机"，而是产品的最终负责人 |
| **Scrum Master（SM）** | 确保 Scrum 流程被正确执行、消除团队障碍、引导会议 | SM 不是"项目经理"，而是教练和仆人式领导者 |
| **Development Team** | 自组织地完成 Sprint 目标，交付可工作的软件增量 | 团队成员不是"资源"，理想规模 5-9 人 |

### 2.2 五大事件

```text
Sprint（时间盒：1-4 周，通常 2 周）
│
├── Sprint Planning（Sprint 开始时，≤ 8 小时 / 月 Sprint）
│   ├── 议题 1：这个 Sprint 能交付什么？（PO 讲优先级）
│   └── 议题 2：如何完成这些工作？（团队拆解任务）
│
├── Daily Scrum（每天 ≤ 15 分钟，同一时间、同一地点）
│   ├── 昨天做了什么？
│   ├── 今天打算做什么？
│   └── 有什么阻碍？
│
├── Sprint Review（Sprint 结束时，≤ 4 小时 / 月 Sprint）
│   └── 向利益相关者演示可工作的软件，收集反馈
│
└── Sprint Retrospective（Review 之后，≤ 3 小时 / 月 Sprint）
    └── 团队内部反思：什么做得好？什么可以改进？下个 Sprint 的行动项
```

::: warning 常见误区
- Daily Scrum 不是"汇报给 Leader"的会议——是团队成员之间的同步，SM 只是旁观引导
- Retrospective 不是"批斗会"——目标是改进流程，不是追责个人
- Sprint Review 不是"演示 PPT"——必须演示**可工作的软件**
:::

### 2.3 三大工件

| 工件 | 内容 | 负责人 |
|------|------|--------|
| **Product Backlog** | 产品的所有需求，按优先级排序，持续演化 | PO |
| **Sprint Backlog** | 当前 Sprint 选中的 Backlog + 任务拆分 | Development Team |
| **Increment** | Sprint 结束时完成的、可交付的软件增量 | Development Team |

## 三、Kanban 方法

### 3.1 Kanban 的核心原则

Kanban 来自精益生产（丰田生产方式），核心是**可视化工作流**和**限制在制品（WIP，Work In Progress）**。

```text
Kanban 看板示例：

  Backlog         开发中 (WIP≤3)     测试中 (WIP≤2)     完成
+------------+   +-------------+   +-------------+   +--------+
| 需求 H     |   | 需求 E [→]  |   | 需求 C      |   | 需求 A |
| 需求 I     |   | 需求 F      |   | 需求 D      |   | 需求 B |
| 需求 J     |   | 需求 G      |   |             |   |        |
+------------+   +-------------+   +-------------+   +--------+
```

### 3.2 WIP 限制的意义

WIP 限制是 Kanban 的核心机制——每个阶段同时进行的任务数量有上限。

**为什么限制 WIP？**
- 降低上下文切换成本（一个人同时做 3 件事 = 频繁切换 = 效率低）
- 暴露瓶颈（"测试中"列满了而"开发中"列有空余 = 测试是瓶颈）
- 减少交付周期（完成一件再开始下一件，比并行多件更快）

### 3.3 Scrum vs Kanban

| 维度 | Scrum | Kanban |
|------|-------|--------|
| **节奏** | 固定时间盒（Sprint） | 持续流动，无固定周期 |
| **角色** | 定义 PO、SM、Team 三个角色 | 无强制角色 |
| **变更** | Sprint 内不允许改变 Sprint Backlog | 随时可以变更优先级 |
| **度量** | 速率（Velocity）——每个 Sprint 完成的故事点 | 交付周期（Lead Time）、吞吐量 |
| **适用场景** | 有稳定团队和可规划工作的产品开发 | 运维、支持类工作，或需求波动大的场景 |

::: tip 实践趋势
绝大多数团队实际使用"**Scrumban**"——以 Scrum 的时间盒和角色为基础，引入 Kanban 的 WIP 限制和可视化看板。
:::

## 四、用户故事

### 4.1 用户故事的标准格式

```
作为 <角色>，
我想要 <功能>，
以便 <价值/目的>。
```

示例：
```
作为 注册用户，
我想要 通过手机验证码登录，
以便 不需要记住复杂的密码也能安全登录。
```

### 4.2 INVEST 原则

好的用户故事应该满足 INVEST 原则：

| 原则 | 含义 | 反例 |
|------|------|------|
| **I**ndependent | 故事之间独立，可以独立开发和测试 | "实现登录"依赖"实现用户注册" |
| **N**egotiable | 故事是可协商的，不是详细需求文档 | 故事中规定了具体的技术方案和 UI 样式 |
| **V**aluable | 对用户或业务有明确价值 | "重构数据访问层"——这是技术任务，不是用户故事 |
| **E**stimable | 团队能够估算故事的工作量 | "研究 AI 推荐算法"——不确定性太高，无法估算 |
| **S**mall | 足够小，能在 1 个 Sprint 内完成 | "构建电商平台"——太大了，需要拆分 |
| **T**estable | 有明确的验收标准，可测试 | "让用户体验更好"——无法测试 |

### 4.3 用户故事拆分技巧

将"史诗"（Epic）拆分为可在一个 Sprint 内完成的故事：

| 拆分策略 | 示例（"用户搜索商品"史诗） |
|----------|---------------------------|
| **按操作拆分** | 关键字搜索 → 分类筛选 → 排序 |
| **按数据变体** | 搜索有库存商品 → 搜索无库存商品 → 搜索下架商品 |
| **按简单/复杂** | 基本搜索 → 高级搜索（多条件组合） |
| **按平台** | Web 搜索 → 移动端搜索 |

### 4.4 验收标准（Acceptance Criteria）

每个用户故事都应该有清晰的验收标准：

```text
故事：作为用户，我想要通过手机验证码登录

验收标准：
- [ ] 输入手机号后，点击"获取验证码"，收到含 6 位数字的短信
- [ ] 60 秒内不可重复获取验证码
- [ ] 输入正确验证码后进入首页
- [ ] 输入错误验证码 3 次后锁定 5 分钟
- [ ] 未注册的手机号提示"请先注册"
```

## 五、Sprint 计划与估算

### 5.1 故事点估算

故事点（Story Point）是**相对估算**单位——不是"小时"或"人天"，而是相对于一个基准故事的工作量。

```text
估算刻度：1, 2, 3, 5, 8, 13, 21...（斐波那契数列）

为什么用斐波那契？
- 任务越大，估算的不确定性越高
- 13 和 21 之间的差距警告团队：这个任务需要拆分了
```

### 5.2 速率（Velocity）

速率 = 团队过去几个 Sprint 完成的故事点平均值。

```text
过去 3 个 Sprint:
  Sprint 1: 完成 20 点
  Sprint 2: 完成 22 点
  Sprint 3: 完成 18 点

平均速率 = (20 + 22 + 18) / 3 = 20 点

下一个 Sprint 的容量计划：20 点
```

::: warning 注意
速率用于**计划**而非**考核**。不能用速率来比较不同团队的产出——每个团队的估算基准不同，点值没有可比性。
:::

### 5.3 Sprint 容量规划

```text
团队：5 人，Sprint 周期：2 周（10 个工作日）

总容量 = 5 * 10 = 50 人天
减去：
  - 会议时间（Daily Scrum 15min * 10 + Planning + Review + Retro）≈ 3 天
  - 技术债务偿还 / 修复线上问题 ≈ 2 天
  - 其他（面试、分享、突发）≈ 2 天
实际可用 = 50 - 7 = 43 人天 ≈ 考虑速率后约 20 故事点
```

## 六、持续改进 —— 回顾会议

### 6.1 回顾会议的结构

回顾会议（Retrospective）是 Scrum 中最重要的会议，是团队自我改进的核心机制。

```text
1. 设定基调（5 分钟）：建立安全氛围——"这是团队的反思，不是追责"

2. 收集数据（10 分钟）：这个 Sprint 发生了什么？
   - 时间线回顾：Sprint 中发生了哪些关键事件？
   - "Start / Stop / Continue"：什么应该开始做 / 停止做 / 继续做？

3. 分析洞察（15 分钟）：为什么发生了这些事？
   - 5 Why 分析法：连续问 5 个"为什么"找到根因

4. 决定行动（10 分钟）：下一个 Sprint 要改变什么？
   - 选择 1-2 个改进项（不要贪多）
   - 明确行动负责人和检查时间

5. 结束（5 分钟）：感谢团队，确认行动项
```

### 6.2 5 Why 分析法

```text
问题：线上出现了一个数据不一致的 bug

Why 1: 为什么数据不一致？
  → 因为两个服务对同一条数据做了冲突的更新

Why 2: 为什么两个服务会冲突更新同一条数据？
  → 因为我们没有定义这条数据的"归属服务"

Why 3: 为什么没有定义归属？
  → 因为在微服务拆分时，数据归属没有被明确讨论

Why 4: 为什么没有被讨论？
  → 因为我们的设计评审 Checklist 中没有"数据归属"这一项

Why 5: 为什么 Checklist 中没有？
  → 因为 Checklist 是两年前定的，之后没有更新过

根因：设计评审流程缺少定期维护机制
改进行动：每季度 Review 并更新设计评审 Checklist
```

::: tip 5 Why 的要点
1. 追问到"可行动的根因"为止——不是"人的错误"，而是"流程的漏洞"
2. 避免"因为他不小心"——这是指责，不是分析
3. 如果 5 Why 后得出的结论是"加一个审批流程"，那很可能是错误的——好的根因导致**减少流程**而非增加流程
:::

### 6.3 改进看板

将 Retro 产生的改进项可视化，避免"每次 Retro 都在讨论同样的问题"：

```text
改进看板：
  ├── 待开始：引入 API 自动化测试框架
  ├── 进行中：建立代码 Review Checklist
  ├── 已完成：统一错误码规范
  └── 已验收：日志规范落地（已在 2 个 Sprint 中验证有效）
```

## ⭐ 面试高频问题

### Q1：敏捷和瀑布模型的核心区别是什么？分别适用什么场景？

**标准答案**：

| 维度 | 瀑布 | 敏捷 |
|------|------|------|
| **需求处理** | 一次性确定，冻结变更 | 迭代演化，欢迎变化 |
| **反馈循环** | 项目尾声才有用户反馈 | 每个 Sprint 获取反馈 |
| **风险暴露** | 集中在后期 | 持续暴露，早期化解 |
| **交付方式** | 最终一次性交付 | 持续增量交付 |

- **瀑布适用**：需求极其明确且不会变化（航天嵌入式软件、核电站控制系统），错误的代价远大于需求变更的成本
- **敏捷适用**：需求不确定或快速变化（互联网产品、创业项目），快速验证比完美实现更重要

**面试金句**："瀑布适合'盖大桥'——需求明确、变更成本高。敏捷适合'做产品'——你不知道用户真正想要什么，需要通过持续交付和反馈来探索。"

### Q2：Scrum 的三个角色分别是什么？各自的职责是什么？

**标准答案**：

- **Product Owner（PO）**：回答"做什么"——管理 Product Backlog 优先级，定义验收标准，是产品的最终负责人
- **Scrum Master（SM）**：回答"如何高效"——确保 Scrum 流程被正确执行，消除团队障碍，引导会议，是教练和仆人式领导者
- **Development Team**：回答"如何做"——自组织地完成 Sprint 目标，交付可工作的软件增量，5-9 人

**关键区分**：PO 不是"提需求的人"，SM 不是"项目经理"，Development Team 不是"被分配任务的资源"。

### Q3：一个 Sprint 中，需求可以变更吗？

**标准答案**：

不能。一旦 Sprint Backlog 在 Sprint Planning 中被确认，**Sprint 期间不允许变更**。这是 Scrum 的核心规则——给团队一个不受干扰的专注期。

但 Sprint 之间可以任意变更——上一个 Sprint 完成后，下一个 Sprint Planning 中 PO 可以根据最新情况重新排列 Backlog 优先级。

::: tip 如果确实需要变更
两种选择：
1. **紧急变更**：PO 可以宣布终止当前 Sprint（极少使用），启动新的 Sprint Planning
2. **等待**：将新需求放入 Product Backlog 顶部，下一个 Sprint 优先处理
:::

### Q4：故事点和人天有什么区别？为什么用故事点估算？

**标准答案**：

| 维度 | 故事点 | 人天 |
|------|--------|------|
| **性质** | 相对估算（相对于基准故事） | 绝对估算 |
| **反映** | 工作量 + 复杂度 + 不确定性 | 仅反映时间 |
| **团队差异** | 不同团队的"1 点"含义不同 | "1 天"理论上跨团队一致 |
| **准确度** | 团队一致时准确度高 | 总是偏乐观 |
| **优势** | 避免"这个任务 3 天就能搞定"的盲目自信 | 直观，非技术人员也能理解 |

**为什么用故事点**：人天估算有锚定效应（"一个页面开发需要 3 天"——但实际上不同的"页面"复杂度可能差 10 倍），且人天估算忽略了复杂度和不确定性。故事点通过相对比较（"这个页面比上次那个复杂 2 倍"）得到更准确的估算。

### Q5：你们团队如何进行 Sprint 计划？

**标准答案**：

1. **PO 提前准备**：PO 在 Planning 前已经按优先级排序好 Backlog，Top 部分有清晰的验收标准
2. **容量确认**：根据团队速率（过去 3 个 Sprint 的平均故事点）和可用人天，确定本 Sprint 容量
3. **讲解与澄清**：PO 逐条讲解顶部 Backlog 的需求和验收标准，团队提问澄清
4. **估算**：团队对每个 Story 进行故事点估算（Planning Poker 或 T-shirt Sizing）
5. **选择与承诺**：团队按优先级从 Backlog 中选取，直到达到容量上限
6. **任务拆分**：将选中的 Story 拆分为技术任务（每个人 < 1 天的粒度）
7. **输出**：Sprint Goal + Sprint Backlog（Story + Task）

### Q6：如果团队速率（Velocity）不稳定，应该怎么办？

**标准答案**：

速率不稳定是正常现象——新团队形成期、需求类型变化、团队成员变动都会导致速率震荡。

应对策略：
1. **使用滚动平均**：用过去 3 个 Sprint 的平均速率，而非上一 Sprint 的速率
2. **分析变异来源**：是估算不准还是外部干扰？——不同原因不同对策
3. **引入缓冲区**：计划时留 10%-15% 的缓冲（处理突发线上问题、技术债务）
4. **关注趋势而非单点**：速率是逐渐上升还是持续下降？趋势比绝对值更重要
5. **拆分大故事**：点数过大的 Story（> 13 点）估算极不准，拆分后稳定性会提升

**根本原则**：速率是**计划辅助工具**，不是考核指标。不要用速率衡量团队效率——这会催生"点值通胀"。

### Q7：Retro 回顾会议如果大家都不发言怎么办？

**标准答案**：这是常见的 Retro 困境，通常是因为心理安全感不足或会议形式单一。

对策：
1. **改变会议形式**：用"写便签再匿名投票"代替公开发言，降低发言门槛
2. **设定基调**：SM 在会议开始时明确"这不是追责会，是改进会"
3. **分组讨论**：5 人以上团队分成 2-3 人的小组先讨论，再汇总
4. **使用数据**：从 Jira 拉数据——"本 Sprint 有 3 个故事被带回了 Backlog，这是为什么？"——用数据引导讨论
5. **Leader 后发言**：Tech Lead 和 SM 先沉默，让团队成员先说，避免"领导定调"

### Q8：敏捷转型中常见的失败原因有哪些？

**标准答案**：

1. **"伪敏捷"**：用了 Daily Scrum 和 Jira 就叫敏捷了，但本质依然是瀑布思维——需求不欢迎变化、不重视反馈
2. **Leadership 不支持**：管理层嘴上说敏捷，实际上还是在要求"保质保量按期交付"，团队没有真正的自治权
3. **忽视工程技术**：敏捷不只是"站会 + 看板"，TDD、CI/CD、重构等工程实践是敏捷的基础
4. **Retro 流于形式**：每次都讨论同样的问题但从不落地改进，丧失信任
5. **团队太大**：超过 9 人的团队沟通成本急剧上升，敏捷实践难以运转

**一句话总结**：敏捷转型失败的最大原因是——**只改了流程，没改变思维**。
