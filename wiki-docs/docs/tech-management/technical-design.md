## 一、技术选型方法论

### 1.1 选型的本质：在约束条件下做最优决策

技术选型不是"哪个技术最好"，而是"哪个技术在当前约束下最优"。面试中，面试官想听到的不是"我用Kafka因为它吞吐量高"，而是"在给定的团队规模、业务场景和时间窗口下，为什么选择A而不是B"。

### 1.2 技术选型五维评估模型

| 维度 | 权重（典型） | 评估要素 |
|------|-------------|----------|
| **业务匹配度** | 30% | 功能满足度、非功能性需求（延迟/吞吐/一致性）、未来演进方向 |
| **团队能力** | 25% | 现有技术栈、学习曲线、社区中的人才供给、内部是否有专家 |
| **运维成本** | 20% | 部署复杂度、监控成熟度、故障恢复能力、是否需要专人维护 |
| **生态成熟度** | 15% | 社区活跃度、文档质量、第三方集成、大厂使用案例 |
| **长期风险** | 10% | 开源协议、商业依赖风险、技术生命周期阶段 |

::: danger 面试高频陷阱
永远不要只说技术优势而忽略成本。面试官最想听到的是你对**权衡**的思考——A方案快但不稳定，B方案稳定但重——你为什么在这样的权衡下做出选择。没有"完美的方案"，只有"在当前约束下合理的方案"。
:::

### 1.3 典型选型案例

**案例：消息队列选型**

某电商团队面临消息队列选型。业务需求是订单状态流转，异步通知下游（库存、物流、通知）。日均订单量50万，峰值约为日均的3倍。

| 候选方案 | 优势 | 劣势 | 团队评价 |
|----------|------|------|----------|
| RabbitMQ | 协议成熟，运维简单，延迟低 | 吞吐量上限约2-3万/s | 团队中有两人用过 |
| Kafka | 吞吐量极高，持久化好 | 运维复杂，延迟稍高，团队没有人用过 | 需额外1-2个月学习 |
| RocketMQ | 阿里出品，中文文档好，事务消息 | 社区门槛降低后变冷 | 学习成本中等 |

最终选择：RabbitMQ。理由：当前并发量（峰值约1800条/s）远未碰到RabbitMQ的吞吐量上限；团队已有的运维经验能降低故障风险；RocketMQ虽好但社区活跃度下降带来长期维护风险。

**面试回答关键**：这个决策不是永久的——如果未来订单量增长到日均500万，RabbitMQ达到瓶颈时，方案应该切换到Kafka。技术选型要留好演进路径。

## 二、架构评审 ADR（Architecture Decision Record）

### 2.1 什么是 ADR

ADR 是一种轻量级的架构决策记录方式——每做一个重要的架构决策，就写一篇短文档记录：**为什么做这个决策、背景是什么、考虑了哪些替代方案、选择了什么、后果是什么**。

### 2.2 ADR 的标准格式

```markdown
# ADR-001: 选择 RabbitMQ 作为消息中间件

## 状态
已接受 / 已废弃（被 ADR-XXX 替代） / 提议中

## 背景
当前系统需要异步解耦订单模块与下游服务（库存、物流、通知）。
日均订单量50万，峰值约为日均3倍。团队4人，2人有RabbitMQ使用经验。

## 决策
采用 RabbitMQ 作为消息中间件。

## 考虑的替代方案
1. **Kafka**: 吞吐量高但运维复杂，团队无经验，学习成本约1-2个月
2. **RocketMQ**: 功能丰富但社区活跃度下降
3. **Redis Stream**: 轻量但持久化能力和消息可靠性不如专用MQ

## 后果
- 正面：快速上手，运维简单，满足当前吞吐量需求
- 负面：当订单量增长到日均500万时需要迁移到Kafka
- 风险：单机RabbitMQ的可用性风险——第一期手动切换，第二期引入镜像队列
```

### 2.3 ADR 的价值与落地

| 价值维度 | 说明 |
|----------|------|
| **防止重复讨论** | "为什么当时选了RabbitMQ？"不再需要去翻聊天记录 |
| **新人上手** | 新人通过读ADR就能理解系统为什么长这样 |
| **决策回溯** | 当约束条件变化时（如订单量翻了10倍），ADR提供了"重新评估"的锚点 |
| **架构治理** | ADR 的"状态"字段让废弃决策有迹可循 |

::: tip 落地经验
ADR 不需要写成论文。每个 ADR 控制在 1-2 页。存放到项目仓库的 `docs/adr/` 目录下，编号递增。不要等到"完美"再写——决策做出后就写，即使不完美也是真实的上下文记录。
:::

## 三、技术文档写作

### 3.1 技术文档的四层分类

```text
第一层：设计文档
  └── 写给谁：架构师、Tech Lead
  └── 什么时候写：做重大技术决策前
  └── 格式参考：ADR、技术方案设计书

第二层：接口文档
  └── 写给谁：调用方开发者
  └── 什么时候写：API 设计完成后、编码前（Contract-First）
  └── 格式参考：OpenAPI/Swagger、Proto 文件 + 注释

第三层：运维文档
  └── 写给谁：运维、SRE、On-Call 值班
  └── 什么时候写：上线前、每次故障后更新
  └── 格式参考：Runbook、故障手册、监控大盘说明

第四层：知识文档
  └── 写给谁：新入职的团队成员
  └── 什么时候写：持续迭代
  └── 格式参考：模块README、技术分享记录、FAQ
```

### 3.2 技术文档写作三原则

1. **读者优先**：先定义"谁在读"，再决定怎么写。给PM看的和给运维看的应该是两篇不同的文档。
2. **场景驱动**：不要罗列"系统有哪些功能"，而是"用户要做X时，系统怎么处理"。
3. **持续更新**：文档的敌人是"过期"。与其写十篇不更新的文档，不如维护三篇永远准确的核心文档。

::: warning 常见错误
"看代码就知道了，不用写文档。"——这是典型的IC思维。作为技术管理者，文档不只是一个参考材料，它更是**团队知识传递的载体**和**架构质量的检查工具**。如果你的设计文档写得一团糟，很可能设计本身也是一团糟。
:::

## 四、RFC 流程

### 4.1 RFC（Request for Comments）的生命周期

```text
提出（Proposal）
  └── 任何人可以提出 RFC，技术管理者把关是否需要正式流程
  │
讨论（Discussion）
  └── 给相关方1-5个工作日的Review时间
  └── 异步评论为主，复杂议题安排同步讨论
  │
决策（Decision）
  └── 技术管理者或架构委员会做出最终决策
  └── 记录"批准/拒绝/需修改"及理由
  │
公示（Announcement）
  └── 在全团队范围内通知决策结果和要点
  │
落地（Implementation）
  └── RFC 转化为具体的技术方案文档（即 ADR）
```

### 4.2 RFC 流程的落地要点

在中小团队（5-15人），RFC流程不需要很重：

- **触发条件**：涉及跨模块的影响、引入新的技术栈或中间件、需要团队达成共识的设计决策
- **评审周期**：普通RFC 2个工作日，重大RFC 5个工作日
- **决策机制**：在中小团队，通常由Tech Lead做最终决策（而非投票制，避免"设计委员会"的效率问题）
- **关键原则**：不同意可以提出替代方案，但一旦决策做出，全团队执行

## 五、风险评估

### 5.1 技术方案的风险矩阵

| 风险类别 | 具体风险 | 概率 | 影响 | 缓解措施 |
|----------|----------|------|------|----------|
| **技术风险** | 新技术栈学习曲线超预期 | 中 | 中 | 安排核心成员先做POC |
| **依赖风险** | 第三方服务不稳定或接口变更 | 高 | 高 | 增加熔断降级、签订SLA |
| **人力风险** | 关键成员离职或请假 | 中 | 高 | 保证每个模块至少两人熟悉 |
| **容量风险** | 实际流量远超预估 | 低 | 高 | 提前准备水平扩容方案 |
| **兼容风险** | 老版本客户端不兼容新接口 | 中 | 中 | 接口版本化，灰度发布 |

### 5.2 应急预案的编写模板

每个重大上线必须有回滚方案（Rollback Plan），且要在上线评审会上重述一遍：

```text
1. 回滚触发条件（什么情况下应该回滚？）
   - 核心指标下降超过X%
   - 错误率超过X%
   - 用户投诉超过X条/小时

2. 回滚步骤（操作人、操作步骤、预估时间）
   - 步骤1: 通过配置中心关闭新功能开关（< 1分钟）
   - 步骤2: 如开关无效，回滚到上个版本的镜像（< 5分钟）
   - 步骤3: 验证核心链路正常（< 3分钟）

3. 数据回滚方案（如涉及数据变更）
   - 是否有数据兼容性保证？
   - 是否需要数据回滚脚本？

4. 回滚后的通知机制
   - 通知相关团队和业务方
```

## ⭐ 面试高频问题

### Q1：请描述你的技术选型方法论。能否举例说明？

**标准答案**：

我的方法论是"约束-选项-评估-决策-记录"五步法：

1. **明确约束**：业务场景、团队能力、时间窗口、预算、现有系统兼容性
2. **列出选项**：至少3个候选方案，包括"维持现状"
3. **多维评估**：用五维评估模型（业务匹配度、团队能力、运维成本、生态成熟度、长期风险）为每个选项打分
4. **做出决策**：在团队内达成共识，记录决策理由
5. **写入ADR**：将决策永久记录，方便未来回溯

实战案例：[分享你的一次实际选型经历，重点说清楚权衡过程]

### Q2：什么是 ADR？你们团队是怎么用的？

**标准答案**：

ADR（Architecture Decision Record）是一种轻量级的架构决策记录。每做一个重要决策，我们就在 `docs/adr/` 下写一篇简短的记录，包含：背景、决策、考虑的替代方案、后果（正面和负面）。

我们团队的原则是：不追求完美，但追求"有记录"。如果每次决策讨论完都没有文档，半年后团队就忘了"为什么这么设计"——ADR 解决了这个"考古"问题。落地时最关键的一点是：决策者自己写 ADR，不要转交给其他人——转交了就会丢失决策的上下文和微妙考量。

### Q3：你怎么判断一份技术方案的质量？

**标准答案**：

我看技术方案时有六个检查点：

1. **目标是否清晰**：第一页能不能说清楚要解决什么问题，ROI是什么
2. **是否有备选方案**：单一方案说明思考不够充分，至少要有两个备选方案和放弃理由
3. **非功能性需求是否覆盖**：不只是"功能能做出来"，还要考虑性能、安全、可观测性、容错
4. **是否画了核心链路图**：一图胜千言，时序图/架构图是技术方案的骨架
5. **是否写了风险点**：有没有主动识别风险？有没有回滚方案？
6. **是否可执行**：能不能拆出3-5个独立的开发任务，每个任务有预估工时

如果有3个以上检查点不合格，我会让作者拿回去重写——这不是不近人情，而是方案质量的底线。

### Q4：你怎么推动 RFC 流程在团队中落地？

**标准答案**：

推动的关键是**先小范围做示范，再逐步扩大**：

1. 我作为 Tech Lead 先写 2-3 篇 RFC 出来，让团队看到"原来是这个格式"
2. 每次评审会上提及："这个决策有没有RFC？"形成习惯
3. 把 RFC 的 Review 过程做成异步的（通过文档评论），不额外开会
4. 降低门槛：RFC 不要求完美——一个核心决策 + 两个备选方案分析就足够
5. 建立正循环：当团队成员发现"上次那个RFC帮我挡掉了PM的无理需求"时，他们自然会愿意写

最难的其实是"什么时候需要RFC"的判断。我的规则是：如果这个决策会影响超过3个人或持续超过6个月，就应该写RFC。

### Q5：一个大项目上线，你怎么做风险评估和应急准备？

**标准答案**：

我的流程分为四步：

1. **上线前评审**：拉上相关方（开发、测试、运维、产品），对照风险清单逐项确认。重点关注数据变更、接口兼容性、第三方依赖变化。

2. **灰度策略**：按用户比例逐步放量（1% → 10% → 50% → 100%），每个阶段观察核心指标至少30分钟。宁可多花半天时间灰度，也不要全量上去后紧急回滚。

3. **回滚方案**：每个上线必须有明确的回滚条件（错误率>X%、延迟>Xms）、回滚步骤（精确到每分钟做什么操作）、回滚验证（回滚后怎么确认系统恢复正常）。

4. **值守机制**：上线后核心开发人员至少值守1小时，确保有异常时能第一时间响应。我个人的原则是：**谁写的代码谁值守，谁做的决策谁负责**——上线不是"扔过墙"，而是责任的延续。

如果面试官追问"有没有上线失败的案例"——一定要诚实说一个，重点放在复盘结论和后续改进上，而不是把锅甩给"测试没测好"或"运维操作失误"。

### Q6：你遇到过技术方案被否决的情况吗？怎么处理的？

**标准答案**：

当然遇到过。有一次我主导的技术方案在评审会上被架构师否决了，理由是"耦合了太多业务逻辑到基础设施层"。

处理方式：第一，先冷静听对方说完理由——不要在情绪上防御，把评审当成改进自己方案的机会。第二，会后我用对方指出的问题重新审视了一遍方案，确实发现有三处耦合可以解耦。第三，我当天就把修改后的方案发回去，附上"感谢指正，修改后的方案解决了XX问题"。

这件事让我学到的是：技术方案被否决不是对你的否定，而是对你工作的**质量检查**。一个管理者如果受不了自己的方案被质疑，那他带团队时也会受不了别人的不同意见。如果你想听到"我的方案从来没被否决过"——那就说明你从来没有提交过有任何风险的方案，这并不是好信号。

### Q7：如何推动团队写好技术文档？

**标准答案**：

这个问题背后折射的是"技术文档谁都不爱写"的现实。我的策略分三步：

1. **降低门槛**：不要要求"完整的设计文档"，而是先用模板驱动——一个标准的方案模板（背景-方案-风险-排期）比一张白纸容易写得多。

2. **绑定流程**：文档不只是"写完看的"，而是和具体流程绑定——例如，没有方案文档不允许进入开发阶段，没有Runbook不允许上线。文档有了真实价值，写作意愿自然上升。

3. **以身作则**：作为技术管理者，你自己的方案和Code Review评论是最好的模板。如果你自己的Code Review永远只是"LGTM"三个字母，就不要指望团队成员写出有深度的设计文档。
