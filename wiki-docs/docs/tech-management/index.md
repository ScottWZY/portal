# 技术管理

> 技术管理是 P7+/技术专家的核心能力领域。从 IC（Individual Contributor）到技术管理者的转变，是职业生涯中最具挑战性的一跃。本模块聚焦技术Leader与技术经理的角色差异、核心职责、以及管理思维的形成路径。

## 一、技术管理者的角色认知

### 1.1 技术Leader vs 技术经理

这是面试最高频的考察点之一。两者并非互斥关系，而是对同一个岗位的不同侧重。关键不是二选一，而是能根据团队阶段和发展需要动态调整比例。

| 维度 | 技术Leader | 技术经理 |
|------|-----------|----------|
| **核心产出** | 技术决策质量、架构设计、技术方案 | 团队效率、人员成长、目标达成 |
| **影响力来源** | 技术声望与判断力 | 职权与资源调配能力 |
| **日常工作** | 编码、Code Review、设计评审、技术选型 | 一对一沟通、绩效评估、资源分配、跨团队协调 |
| **衡量指标** | 系统可用性、技术债趋势、交付质量 | 团队满意度、人员留存率、业务交付及时率 |
| **典型困境** | 被业务需求淹没，没有时间做深度技术 | 脱离代码后被团队质疑技术能力 |
| **理想比例** | 40%-60% 技术工作（编码 + 评审 + 设计） | 60%-80% 管理工作（人员 + 流程 + 协调） |

::: tip 面试金句
"技术Leader靠'影响力'驱动团队，技术经理靠'权力'管理团队。P7+必须两者兼备——没有技术深度的经理会丧失团队的信任，没有管理意识的Leader会让团队陷入混乱。"
:::

### 1.2 核心职责矩阵

技术管理者需要同时在三个维度上发力：

```text
个人贡献（IC）
  ├── 关键模块开发（占比逐渐降低，但绝不归零）
  ├── 技术难点攻关
  └── 技术方案设计

团队杠杆（Team Leverage）
  ├── Code Review与质量保障
  ├── 技术评审（RFC / ADR 流程）
  ├── 新人培养与导师制
  └── 技术分享文化建立

组织影响（Org Impact）
  ├── 跨团队技术协作与标准制定
  ├── 技术规划与技术债管理
  └── 招聘与人才梯队建设
```

这三个维度的权重分配是动态的：团队早期更偏IC，成长期加重团队杠杆，成熟期向组织影响倾斜。面试中常见的问题是"你现在每天的精力怎么分配？为什么这样分配？"回答的关键是**解释分配背后的逻辑**，而非仅仅报一个比例数字。

### 1.3 技术管理者的常见误区

| 误区 | 表现 | 后果 |
|------|------|------|
| **过度关注编码** | 接手所有关键模块，不信任团队 | 团队成长停滞，自己成为瓶颈 |
| **完全脱离编码** | "我现在就是管人的，不用写代码" | 技术判断力下降，团队技术方向失控 |
| **只做上传下达** | 把业务需求直接转发给团队 | 丧失技术Owner意识，沦为"传话筒" |
| **忽视一对一沟通** | 只在有问题时才找团队成员谈话 | 无法建立信任，问题发现总是太晚 |

## 二、从IC到管理的转变

### 2.1 三个核心转变

**第一个转变：从"我的代码"到"我们的系统"**

IC时代，你为单个模块的质量负责；成为技术管理者后，你需要为整个系统的架构健康度、技术方向、交付节奏负责。这意味着你要放下"这块必须按我的方式写"的执念，学会通过设计评审和编码规范来确保质量，而不是亲自重写。

在面试中，面试官会这样考察："你接手了一个低质量的系统，没有文档、测试覆盖率不到10%。你第一个月会做什么？"

::: tip 标准回答框架
1. 先不急于重构，花两周时间通过监控和日志建立对系统的感性认知
2. 梳理核心链路（用户最常走的那条路径），优先为这些链路补测试
3. 识别最大风险点（全局变量、硬编码配置、未处理的异常），逐个建立防护
4. 建立技术债看板，制定分期偿还计划，让团队对齐优先级
5. 在修复中沉淀文档——每修一个问题都更新对应模块的README
:::

**第二个转变：从"我更快"到"团队更快"**

IC时代，你的效率体现在个人产出上；管理者的效率体现在团队产出上。你需要从"自己做完"转变为"让团队做完"。

实际案例参考：一个需求，你写要3天，新人写要10天。短期看你自己写更快；长期看如果一直自己写，你永远是瓶颈，新人永远不会成长。正确做法是：定义好接口与验收标准，拆出新人能独立完成的部分，用Code Review作为教学机会，让10天逐步变成7天、5天、3天。

**第三个转变：从"确定性执行"到"模糊性决策"**

IC时代，需求是明确的，你负责的是"怎么做"；技术管理者的核心挑战是面对不确定——选择哪个方向、投资哪个技术栈、承接还是拒绝哪个需求。这些决策往往信息不充分、后果难以预判，但必须由你来做。

### 2.2 转变期的典型挑战

入职新公司或刚晋升时，前3个月定生死：
- **第一个月**：理解上下文（业务、系统、团队），不要急于做决策。宁可被认为"慢热"，也不要因为不了解历史而做出错误决策。
- **第二个月**：找到1-2个快速见效的机会（修复一个影响所有人的构建慢问题、补充一份核心模块的文档、优化一个高频慢查询），建立信任。
- **第三个月**：开始制定中长期规划，正式驱动改变。

## ⭐ 面试高频问题

### Q1：技术Leader和技术经理有什么区别？你觉得自己是哪一种？

**标准答案**：

核心区别在于驱动力来源：技术Leader靠技术声望与判断力驱动团队，技术经理靠职权与资源调配管理团队。两者不是非此即彼——P7+要求两者兼备。我个人当前的定位是"70%技术Leader + 30%技术经理"（或反之，根据实际经历给出合理比例和理由）。

判断标准：如果我离开这个团队半年后，团队是否能维持当前的技术水准？如果不能，说明我在管理层面做得不够——过度依赖个人能力而非体系。如果能，说明我建立的分工、规范、评审机制在起作用。

### Q2：你从IC转管理过程中最困难的转变是什么？

**标准答案**：

最困难的是**从"自己的代码质量"到"团队的代码质量"的心态转变**。IC时我对自己的代码有完全的掌控力；转为管理者后，我必须接受团队成员的代码风格和能力差异。我花了很长时间学会：与其自己重写不完美的代码，不如花同样的时间在Code Review中给出有建设性的反馈，让对方成长。

另一个挑战是**碎片化工作模式**。IC时代我能连续4小时沉浸编程；管理者的一天被各种上下文切换打碎——站会、一对一、线上问题、跨团队协调。我学会了把深度工作时间固定在上午2小时（Slack/飞书设为免打扰），并坦然接受下午是碎片化的。

### Q3：作为技术管理者，你如何做技术决策？能否举个例子？

**标准答案**：

我的决策框架是"约束-选项-权衡"三步法：

1. **明确约束**：时间、人力、现有系统、团队能力边界
2. **列出可选项**：至少3个方案，包括"什么都不做"
3. **按权重评估**：从维护成本、团队学习成本、性能、扩展性、与现有系统的匹配度五个维度打分

实战案例：某次我们团队面临缓存方案选型——Redis集群 vs 本地缓存 vs 两层缓存。约束条件是我们只有4个后端开发（没有专职运维），QPS峰值约5000，但对数据一致性要求较高。最终选型本地缓存 + 数据库双写方案——因为团队运维Redis集群的成本远大于本地缓存的性能差距。这个决策的关键在于：**技术选型不是选最好的技术，而是选最适合团队当下阶段的技术。**

### Q4：你的团队中有一个技术很强但不服管理的成员，你怎么处理？

**标准答案**：

首先区分"不服管理"的原因：
- 如果是"觉得我技术不如他"：坦然承认，邀请他在他擅长的领域担任Owner，触发其主人翁意识。技术管理者不需要是最强的，但需要有识人用人的能力。
- 如果是"不认同流程但说不出更好的替代方案"：让他用一次实验来证明更好的方案——给一周时间和一个低风险模块，如果能证明新流程更好，全团队推广。
- 如果是"对什么都不满，但不愿主动改进"：一对一沟通明确期望——技术强是优势，但团队需要的是**有建设性**的影响力，不是纯批判。如果持续无法对齐，需要上升到绩效层面。

### Q5：你如何平衡技术和管理的时间分配？

**标准答案**：

没有固定的比例，关键是根据团队阶段动态调整。我刚带团队时，编码时间约50%；当团队稳定后降低到约20%-30%。具体策略：

- **上午为深度技术时间**（2-3小时免打扰），用于设计评审、关键模块Code Review、技术难点攻关
- **下午为管理时间**：一对一沟通、跨团队协调、招聘面试
- **每周至少保留一个完整下午进行编码**，这是维持技术手感的最低保障线
- **编码内容的选择**：优先选择团队没有人愿意做的脏活累活（工具链改进、CI优化），既贡献力量又避免和团队抢"有意思的功能"

### Q6：如果重新选择，你还会从IC转管理吗？

**标准答案**：

会。原因是：IC让我深度理解了软件，管理让我深度理解了人。两种经历结合，让我能从"做事"和"用人"两个维度同时思考问题，这是单一角色无法获得的视角。

但同时要真诚地说：这条路并不适合所有人。如果你痛恨开会、人际关系让你精疲力竭、你对技术细节有完美主义执着——留在IC路线上你会更快乐、也更有成就。公司需要两条路径并存，关键是诚实面对自己。
