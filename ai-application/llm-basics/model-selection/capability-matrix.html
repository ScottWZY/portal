5.3 决策指标 ​

  • 准确率优先：如果错了后果严重 → 选准确率最高的
  • 成本优先：如果错了可以人工纠正 → 选性价比最高的
  • 延迟优先：如果要求响应快 → 选小模型+缓存

面试高频题 ​

Q1: 如何科学地评估大模型的能力？常用的评测基准有哪些？各有什么局限性？ ​

详细答案： 我们做模型选型的时候从不只看一个 benchmark 就拍板，因为每个基准都有盲区。MMLU 测 57 个学科的多选题，覆盖面广但对中文理解反映不足——Qwen 在 CMMLU 中文版上得分 86.8%，MMLU 上只有 88%。HumanEval 测 Python 函数补全，和真实项目开发差太远——我们做过最真实的 SWE-bench 对比，Claude Sonnet 得分 72.7%，HumanEval 得分 90.8%，两个数据对应的能力维度完全不同。

最大问题是数据污染——很多 benchmark 的测试题已经混入训练数据里了，高分可能是"背答案"而不是"理解"。我们更信赖自己的评估集：从线上采样 50 个真实问题，用 RAGAS + 人工盲评双重验证。外部基准当海选初筛，内部评估集当最终决策依据。

Q2: 哪个模型的中文能力最好？为什么国产模型在中文评测上领先？ ​

详细答案： 从实际体验看，Qwen3-Max 和 DeepSeek R1 的中文能力确实比 GPT-4o 和 Claude 强一个档次。用户说"这破手机烫得跟烤红薯似的"，Qwen 会理解这是抱怨并关联退货流程，GPT-4o 倾向于理解为"手机发热问题需要技术支持"。差别的原因是训练语料——国产模型中文语料占比高，而且包含了大量中文互联网内容、成语典故、草根表达，这些在国际模型的训练数据里几乎为零。

但国产模型中文强不意味着所有中文任务都该用它。我们做过评估——客服 FAQ（纯中文场景）用 DeepSeek 比 GPT-4o 好，但涉及中文法律条文分析或者需要深入推理的长文档，Claude 的理性和逻辑性反而更好。实用建议是中文项目首选国产模型，如果效果不达预期再用国际模型做对比。

Q3: Claude 在代码能力上为什么领先？SWE-bench 是什么？如何解读 SWE-bench 得分？ ​

详细答案： SWE-bench 是目前最接近真实软件开发能力的评测基准——它从 GitHub 上拉了 2294 个真实的 Python issue，要求模型在完整代码仓库里定位 bug 并生成修复补丁，通过率就是补丁能通过该项目所有测试用例的比例。这和 HumanEval 的"补全一个函数"完全不是一个难度——SWE-bench 需要模型理解多文件上下文、定位问题、修改代码、验证正确性四个步骤全通。

Claude Sonnet 4.6 以 72.7% 领先，比 o3 的 55% 和 GPT-4.1 的 42% 高出一大截。原因是 Anthropic 在 Agent 编程和工具使用上的研发投入很深——Claude 的 Computer Use 能力、长上下文理解和多步推理让它在这个全流程任务中特别适配。

但要提醒的是：SWE-bench 高分不代表在所有编程任务上都更好。我们做代码审查辅助用 Claude，但简单的代码补全用 DeepSeek V4 就够了，效果差不多但成本只有 Claude 的 1/15。基准永远是参考，不代表你的具体场景。

Q4: 在你的项目中如何对比和选择模型？说说完整的方法论和考量维度。 ​

详细答案： 我们选型有一套固定流程，五步走。第 1 步明确需求——先写清楚这个功能的核心任务是什么、准确率要求、延迟容忍度。第 2 步初筛——基于公开基准和社区口碑圈 3-5 个候选。第 3 步建评估集——从线上采样 50 个真实问题，人工标注关键信息点。第 4 步自动化评估——并行调用所有候选模型跑评估集，记录 RAGAS 分数、Token 消耗、延迟、费用。第 5 步综合决策——准确率、成本、延迟三维打分，加权排序。

实际选型中我们每次都会做成本推演：日调用量 x 单次成本 = 月费。如果月费超过预算，就得重新调整模型组合。最终考量维度就是四个：能力够不够、能不能控制成本、延迟是否在 SLA 内、数据合规有没有风险。做简单评估矩阵，数据和钱包说话。

Q5: 推理模型（如 o3、DeepSeek R1）和通用模型（如 GPT-4o、DeepSeek V3）的核心区别是什么？什么场景下需要推理模型？ ​

详细答案： 最简单的理解：通用模型是"快问快答"，推理模型是"三思后答"。通用模型收到问题直接生成下一个 token，推理模型会先在内部跑一段 Chain of Thought（思维链），分步推导、自我验证，然后再输出最终答案。你可以把推理模型想象成草稿纸上打好草稿再誊写答案。

我们用推理模型的场景很明确：数学计算、复杂代码调试、多条件决策。比如用户问"我有两张优惠券，一张满 200 减 50，一张打八折，买两件商品分别 120 和 180，怎么用最划算"——这个如果给通用模型直接答，大概率出错。DeepSeek R1 会先分步算出四个组合方案再比较，准确率高很多。

代价就是推理模型烧的 Token 多（思维链要钱）、延迟高、而且简单问题也会"过度思考"浪费资源。我们的策略：加一个路由——先用轻量级模型判断问题是否需要推理，简单问题直接走通用模型，需要多步推导的才切推理模型。DeepSeek R1 的性价比是推理模型里最高的，价格仅 o3 的 1/37 左右，我们的推理场景基本全用 R1。
