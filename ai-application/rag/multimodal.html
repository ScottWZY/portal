多模态 RAG 完整指南（Java 后端开发者视角） ​

面向后端开发者的系统化多模态检索增强生成指南。假设你有传统 RAG 经验，现在需要把图片、表格、扫描件也纳入知识库。

1. 概述 ​

多模态 RAG（Multimodal RAG）是传统 RAG 的扩展：不仅索引文本，还索引图像、表格、图表、扫描 PDF 等非文本模态。

解决的痛点：

  • 传统 RAG 用 OCR 把 PDF 转文本后向量化，表格结构丢失、图片信息完全忽略。
  • 用户问"这个架构图中的模块 A 是什么"时，传统 RAG 无法回答。
  • 财务报表、学术论文中的图表数据无法被检索和利用。

2. 多模态 Embedding ​

传统文本 Embedding（如 text-embedding-3-small）只看文本语义；多模态 Embedding 模型能将文本和图片映射到同一个向量空间，实现"以文搜图"和"图文互搜"。

模型参数量特点推荐场景
CLIP (ViT-B/32)150MOpenAI 出品，图文对齐开创者通用零样本分类、简单图文检索
Jina CLIP v1/v2223M支持 89 种语言，文本+图片双塔中文/多语言场景
SigLIP400M+Google 出品，sigmoid loss 训练高精度图文检索
Chinese-CLIP~200M中文优化版 CLIP纯中文图片检索
ColPali (PaliGemma)3B直接用视觉语言模型做检索文档视觉检索（第 7 节详解）

选型建议（Java 后端视角）：

  • 通用场景优先选 Jina CLIP v2（多语言 + 开源 + 有 REST API）。
  • 纯中文场景用 Chinese-CLIP 或 BGE-VL。
  • 对 PDF 扫描件精度要求高时用 ColPali，但推理成本较高。
  • Java 端通过 HTTP 调用 Embedding 服务即可，无需 Python 运行时。

3. 表格提取 ​

表格是 RAG 中最容易丢失信息的模态。OCR 后的表格变成散落数字，ChatGPT 无法理解其结构。

7. 实战：用 ColPali 做文档视觉检索 ​

ColPali 是一种颠覆性思路：不再先 OCR 再向量化，而是直接用视觉语言模型（PaliGemma）对 PDF 页面截图做检索。

7.1 为什么 ColPali 更好？ ​

传统方案ColPali 方案
PDF → OCR → 文本块 → EmbeddingPDF → 截图 → ViT 编码 → 晚交互检索
OCR 误差累积无 OCR，直接视觉理解
表格、图表信息丢失页面视觉全保留
流水线长，工程复杂端到端，工程简单

7.2 ColPali 代码示例 ​

python
# 使用 ColPali 做文档视觉检索
from colpali_engine.models import ColPali, ColPaliProcessor
import torch

model = ColPali.from_pretrained(
    "vidore/colpali-v1.2",
    torch_dtype=torch.bfloat16,
    device_map="cuda"
)
processor = ColPaliProcessor.from_pretrained("vidore/colpali-v1.2")

# 将 PDF 页面渲染为图片后编码（需先将 PDF 转图片）
images = load_page_images("document.pdf")  # 每页一张图
page_embeddings = []
for img in images:
    inputs = processor.process_images([img]).to("cuda")
    with torch.no_grad():
        emb = model(**inputs)  # 输出 [1, N_patches, dim] 的 patch 级别向量
    page_embeddings.append(emb)

# 查询也经过同一模型编码
query_inputs = processor.process_queries(["Q3 营收增长率是多少？"]).to("cuda")
query_emb = model(**query_inputs)

Java 对接要点：

  • ColPali 推理必须用 Python（GPU），Java 侧只负责调用服务 API。
  • 向量存储需支持多向量（multi-vector），如 Milvus 2.4+ 或 Qdrant。

8. 常见陷阱 ​

  1. 直接用 OCR 全文代替结构化提取：OCR 把表格拆成散落文字，LLM 无法推理行/列关系，必须先用 Table Transformer + 结构化输出。
  2. 图片只用 Caption 不保留原图：Caption 可能有信息损失（如颜色、数值），检索阶段应保留原图多模态向量。
  3. 混用不同 Embedding 空间做融合：文本 Embedding 和图片 Embedding 不在同一向量空间，融合前必须做空间对齐或使用统一多模态模型。
  4. 忽略查询改写：用户问"这个图表说明了什么"时，需要把"这个"替换为具体上下文，否则检索无意义。
  5. Chunk 策略不当：PDF 页面级别的 chunk 太大，会稀释图片和表格的语义信号，建议图片/表格独立 chunk。
  6. ColPali 的存储成本：每个 page 存 1030 个 patch 向量（128 维），存储是传统方案的 10-20 倍，需评估成本。

9. 面试高频题 ​

Q1: 传统 RAG 和多模态 RAG 的核心区别是什么？ ​

详细答案： 我们做过一个保险产品手册的多模态 RAG，对比过传统 RAG 效果，差别非常明显。传统 RAG 把整个 PDF 跑 OCR 转文本，图片直接忽略，表格转成乱码文字——产品手册里有产品对比表、赔付比例表、流程示意图，这些关键信息传统 RAG 要么丢了要么错了，用户问"三者险50万和100万保费差多少"，返回结果根本找不到正确数字。多模态 RAG 不一样，它把文本、表格、图片分开处理：表格提取成结构化 Markdown，图片存原图同时生成描述，整个检索能覆盖所有信息。我们上线后，产品手册问答准确率从 55% 提到了 82%，核心差异就是能处理非文本信息了。

Q2: 表格怎么在多模态 RAG 中正确处理？ ​

详细答案： 我们最开始踩了个大坑——直接把整个 PDF 扫成文本，表格就变成一行行散落文字，比如产品对比表变成"产品A 50万 1000元 产品B 100万 1500元"，完全没了行列关系，LLM 根本读不懂行和列对应哪个产品。后来我们改成三步处理：先用 Table Transformer 把页面上所有表格区域检测出来，裁剪下来，然后用 PaddleOCR 把每个单元格的文字识别出来，再重建行列结构输出 Markdown，最后整个 Markdown 表格扔给文本 Embedding 模型。

我们测过，结构化输出之后，表格问答准确率从 40% 直接升到了 85%。这里还有个小经验：复杂表格（比如有合并单元格、多层表头）输出 HTML 比 Markdown 清楚，简单表格（比如两列三行）用 Markdown 就行。我们现在用 Unstructured 的 hi_res 策略，自动检测+提取+输出，一步到位，省心。

Q3: CLIP 和 ColPali 的检索机制有何不同？ ​

详细答案： CLIP 和 ColPali 代表了两种完全不同的检索思路。CLIP 是双塔架构，图片和文本各编码成一个全局向量（比如 768 维），检索时直接算余弦相似度，很快很省存。但代价是一张复杂的 PDF 页面被压缩成一个向量，细节全丢了，比如页面里的一个小表格数字根本体现不出来。ColPali 不一样——它用 PaliGemma 视觉模型把每页 PDF 切成 1030 个 patch，每个 patch 独立编码，检索时 query 和每个 patch 逐一计算相似度取最大（MaxSim），这就保留了像素级的定位精度。

我们试过用 CLIP 和 ColPali 分别做保险条款页面的检索，CLIP 的准确率大概 65%，ColPali 能干到接近 90%。但 ColPali 代价确实大——存储是 CLIP 的 10 倍，我们 5000 份条款页面如果用 ColPali 存储要占将近 50GB，CLIP 只要 5GB。所以最后的折中：全量索引用 CLIP 做初筛，高价值页面（费率表、理赔流程图）用 ColPali 做精排，兼顾成本和精度的平衡。

Q4: 多模态 RAG 中怎么处理图文混合的 PDF 页面？ ​

详细答案： 我们项目里的产品手册就是典型的图文混排页面——一段文字说明，旁边一张流程图，下方一个对比表格，三者是关联的，但得分别处理。我们用 Unstructured 做页面分解，自动把一页拆成"文本块""表格块""图片块"三类元素。文本块直接按 512 tokens 切分后 Embedding；表格块转 Markdown 后 Embedding；图片块分两路——一路用 Jina CLIP 做多模态向量，一路用 BLIP-2 生成自然语言描述然后做文本向量存储，这样文本检索和图片检索都能召回这张图。

检索时也是多路并行——文本库捞 Top-10，表格库捞 Top-5，图片库捞 Top-5，三路结果用 RRF 融合。融合后送给多模态 LLM（我们现在用的是 Qwen-VL-Max，中文效果好而且支持图片输入），LLM 同时读上下文文本和查看图片，生成最终答案。这里有个坑：不同 Embedding 模型的向量空间不统一，文本 Embedding 和图片 Embedding 不能直接放一起算相似度，必须分开检索再用 RRF 基于排名融合，而不是基于分数融合。

Q5: 多模态 RAG 的评估指标和传统 RAG 有什么不同？ ​

详细答案： 我们评估多模态 RAG 是在传统 RAG 的基础上，又加了几个多模态专属指标。传统的 Recall@K、Faithfulness 还是要评估，这些是基础，但多模态有几个额外的点必须测。第一是表格问答准确率——给定用户问题，你得看 LLM 读对了行和列上的数字没有。我们在 50 份保险产品表格上做评估，一开始准确率只有 40%，后来改成结构化提取，准确率干到了 85%。第二是图片检索准确率——用户问"理赔流程图在哪里"，你得判断检索到的图片是不是正确的流程图，这和文本检索准确率分开评估。第三是图表数值理解准确率——用户问"Q3 同比增长几个百分点"，得看你读对了柱状图上的数字没有。

我们一开始就是用人工评估，找了 50 道典型题（20 道文本、15 道表格、15 道图片/图表），每次改动全量跑一遍看准确率涨了没涨。业界现在有 ViDoRe 这种基准数据集，我们也拿来跑过 baseline 比一比。我的体会是，多模态 RAG 评估比纯文本难多了，很多时候最终还是得靠人工评审，毕竟数值错一个点结果就完全不对。

小结： 多模态 RAG 不是简单加一个 CLIP 模型就完事，它是一套从文档解析到多路召回到多模态生成的系统工程。Java 后端开发者的核心任务不是写 Python 模型代码，而是设计好服务边界、选对工具组合、控制好成本。
