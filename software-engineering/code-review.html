一、Code Review 的核心价值 ​

Code Review 不只是"找 bug"。它的价值是多层次的：

价值层次说明
缺陷预防在代码合入前发现逻辑错误、边界条件遗漏
知识传递让团队成员了解代码变更，消除"单点知识"风险
代码一致性确保代码符合团队规范，风格统一
设计改进审查中讨论架构和设计决策，提升整体设计质量
团队成长新人通过阅读他人代码和审查反馈快速成长
集体拥有感代码不再是"某个人的代码"，而是"团队的代码"

面试金句

"Code Review 最重要的价值不是找 bug，而是知识传递和代码一致性。Bug 可以用测试找，但知识孤岛和风格分裂只能靠 Review 解决。"

二、Code Review 最佳实践 ​

2.1 Review 的粒度 ​

一次 Review 的代码量不应超过 400 行。

研究表明，当 Review 代码量超过 400 行时，缺陷发现率急剧下降。人的注意力是有限的资源。

现实问题

大厂面试常问："如果一个 PR 有 2000 行变更，你会怎么 Review？"

  • 标准回复：请提交者拆分为多个逻辑独立的 commit / PR
  • 如果无法拆分：分多次 Review，每次关注一个核心模块，做好笔记
  • 底线：不对超过 400 行的 PR 直接 Approve

2.2 Review 的节奏 ​

指标建议值原因
Review 响应时间< 4 小时（工作时间）避免上下文切换成本——提交者等太久会忘掉细节
单次 Review 时长< 60 分钟超过 60 分钟注意力下降，Review 质量降低
Review 频率每天 2-3 次固定时间比"随时 Review"更高效，减少中断

三、Code Review 审查清单 ​

3.1 功能性 ​

  • [ ] 代码实现了需求描述的全部功能吗？
  • [ ] 是否处理了边界条件？（null、空集合、0、负数、超大值）
  • [ ] 是否考虑了并发场景？（多线程、分布式）
  • [ ] 错误处理是否完整？（异常不会被吞掉）

3.2 可读性与可维护性 ​

  • [ ] 命名是否清晰、自解释？（不需要注释才能理解）
  • [ ] 函数是否足够短小？（< 40 行，最好 < 20 行）
  • [ ] 是否存在重复代码？（同一逻辑出现 2 次以上）
  • [ ] 圈复杂度是否过高？（> 15 需要讨论是否重构）
  • [ ] 是否存在"神奇数字"？（应该定义为常量）

3.3 设计 ​

  • [ ] 是否符合 SOLID 原则？尤其是单一职责和开闭原则
  • [ ] 类和模块的职责是否清晰？
  • [ ] 是否存在不必要的耦合？
  • [ ] 新增的抽象是否合理？（不过度设计，也不缺少必要抽象）

3.4 安全 ​

  • [ ] 是否有 SQL 注入风险？（是否使用了参数化查询）
  • [ ] 用户输入是否经过校验和过滤？
  • [ ] 敏感信息（密码、Token）是否可能被日志输出？
  • [ ] 权限校验是否完整？

3.5 性能 ​

  • [ ] 是否存在不必要的数据库查询？（N+1 问题）
  • [ ] 循环内是否有耗时操作？（数据库调用、远程 RPC）
  • [ ] 大对象是否及时释放？
  • [ ] 是否有潜在的内存泄漏？

3.6 测试 ​

  • [ ] 是否有足够的单元测试？（核心逻辑覆盖率 > 80%）
  • [ ] 测试是否覆盖了主要分支和边界条件？
  • [ ] 测试是否独立？（不依赖执行顺序）
  • [ ] Mock 使用是否合理？

四、Code Review 文化建立 ​

4.1 心理安全 ​

Code Review 是技术活动，但本质上是人与人的沟通。心理安全是有效 Review 的前提。

原则反例正例
对事不对人"你这样写太蠢了""这个循环内调用 RPC 有性能风险，建议先批量查出来"
提问而非命令"这里必须改成 X""这里如果用 X 方案会不会更好处理并发？你有什么考虑？"
认可好的部分只提问题不提优点"这个错误处理写得很清晰 👍 有个小建议..."
接受被反驳"我是 Senior，听我的""理解你的思路了，那我们保留你的方案，加个注释说明"

4.2 Review 意见分级 ​

使用标签区分意见的严重程度：

标签含义处理
blocker必须修改才能合入（有 bug、安全漏洞）必须修，不修不 Approve
suggestion建议修改，但当前写法也能接受提交者自行判断是否采纳
question提出问题，寻求解释或讨论回答即可，不需要修改代码
praise值得肯定的地方——

4.3 建立 Review 文化的关键实践 ​

  1. 从 Lead 开始：技术 Leader 先接受 Review 并开放讨论，以身作则
  2. Review 速度优先于 Review 深度：快速回复（即便只是"Looks good, will review in detail later"）比深度 Review 更能建立信任
  3. 新人友好：对新人代码的 Review 侧重教学而非批评，多用 question 而非 blocker
  4. 数据驱动：统计 Review 数据（Review 覆盖率、平均响应时间、缺陷拦截率），用数据说服团队

文化建立的核心

Code Review 文化的核心不是"找出别人的错误"，而是**"我们一起让代码变得更好"**。审查者和提交者是一边的，对立面是"潜在的 bug 和技术债务"。

五、工具辅助 ​

5.1 自动化检查（CI 流水线集成） ​

人工 Review 应该聚焦在设计、逻辑、可读性等机器无法判断的维度。以下内容应该在 CI 阶段自动检查：

检查项工具（Java 生态）建议配置
代码风格Checkstyle / SpotBugs团队统一配置，禁止个人修改
静态分析SonarQube质量门禁：新增代码无 Blocker/Critical
圈复杂度SonarQube / PMD方法级别 > 15 报警，> 20 拦截
测试覆盖率JaCoCo + SonarQube增量代码覆盖率 > 80%
安全扫描OWASP Dependency-Check有高危漏洞时自动拦截

⭐ 面试高频问题 ​

Q1：你认为好的 Code Review 是什么样的？ ​

标准答案：好的 Code Review 具有四个特征：

  1. 及时：提交 PR 后 4 小时内得到回复
  2. 聚焦：每次 Review 不超过 400 行，审查者能深度理解
  3. 有建设性：feedback 区分 blocker（必须改）和 suggestion（建议改），对事不对人，既有批评也有认可
  4. 有来有回：审查者提问，提交者回答，形成技术讨论而非单向命令

一句话总结："好的 Code Review 让提交者愿意再提 PR，不好的 Code Review 让提交者尽量避免提 PR。"

Q2：Code Review 时你最关注哪些方面？优先级如何排序？ ​

标准答案：

  1. 正确性（最高优先级）：逻辑是否对？边界条件是否处理？是否存在并发问题？
  2. 安全性：是否有注入风险？敏感信息是否泄露？权限是否正确？
  3. 设计：是否符合设计原则？职责是否清晰？是否存在不必要的耦合？
  4. 可读性：命名是否清晰？方法是否过长？是否存在魔法数字？
  5. 性能：是否有明显的性能问题（N+1、循环内 RPC）？
  6. 测试：核心逻辑是否有测试覆盖？

优先顺序的逻辑：Bug > 安全 > 设计 > 可读性 > 性能 > 测试。Bug 和安全是底线，设计和可读性是长期价值，性能和测试是锦上添花。

Q3：如何处理 Code Review 中的分歧？ ​

标准答案：

  1. 先听对方解释：对方的方案可能有你不知道的上下文约束
  2. 用数据而非权威：不是说"我是 Senior 所以听我的"，而是"这个方案在 XX 场景下有 3 倍性能差距，这是 benchmark 结果"
  3. 对齐原则：回到团队的设计原则和编码规范——原则不一致的先对齐原则
  4. 升级决策：如果 15 分钟内无法达成一致，拉第三方（Tech Lead / 架构师）参与决策
  5. 记录决策：无论结果如何，把决策和理由写在 PR 评论中，方便未来回溯

Q4：你们团队的 Code Review 流程是怎样的？有什么经验教训？ ​

标准答案（候选回答框架）：

流程：提交者自审 → 提 PR（写好描述 + 关联 Issue）→ 指定 1-2 位审查者 → 审查者在 4 小时内 Review → 提交者处理反馈 → 审查者确认 → Approve + Merge。

经验教训：

  1. PR 越小越好：曾经有 PR 2000+ 行，Review 了 3 天，效率极低。后来规定 > 600 行必须拆分
  2. 自动化先行：曾经 30% 的 Review 评论是格式问题，接入 Checkstyle + CI 后这类评论降为零
  3. 新人教育：新人前三个 PR 安排 Senior 做"教学式 Review"，每个 blocker 附带解释和正例代码

Q5：如何看待"Code Review 拖慢交付速度"的观点？ ​

标准答案：

  • 短期确实会慢：Review + 修改反馈需要额外时间（通常 +20%-50% 的代码合入时间）
  • 长期反而更快：Review 减少 bug 返工（修一个线上 bug 的成本是 Review 的 10 倍以上），Review 让团队成员都了解代码（减少"只有一个人懂"导致阻塞的情况）
  • 关键在于优化流程：如果 Review 太慢，问题不在 Review 本身，而在流程——PR 太大、审查者太少、Review 时间不固定

必须强调

Code Review 不是可选项——Google、Microsoft 的研究都表明，Code Review 是发现 bug 最有效的手段，比测试更有效。减少 Review 来提速，等于透支未来来支付现在。

Q6：如何处理遗留代码没有测试、无法安全重构的情况？ ​

标准答案：

  1. 先加特征测试（Characterization Test）：不测试"正确的行为"，只测试"当前的行为"——给现有代码拍一张快照
  2. 从最外层开始：先写集成测试（对接口级别），再逐步深入
  3. 识别接缝（Seam）：找到可以安全修改的点——一个类、一个方法、甚至一个 if 分支
  4. 绞杀者模式：新建模块替代旧模块，而不是在旧代码上修修补补
  5. 每次只改一点点：每次重构后运行特征测试确保行为不变

面试金句："重构的黄金法则——在没有测试的情况下，你是考古学家不是外科医生。你要小心翼翼地挖掘、记录，而不是大刀阔斧地切割。"

Q7：如何衡量 Code Review 的效果？ ​

标准答案：

指标衡量方式
缺陷拦截率Review 发现的问题数 / (Review 发现 + 上线后发现)
Review 覆盖率被 Review 的 PR 数 / 总 PR 数
Review 响应时间PR 提交到首次 Review 的时间中位数
代码质量趋势SonarQube 的代码质量评分变化
知识分布每个模块有多少人 Review 过相关 PR

核心指标是缺陷拦截率——目标是 > 60%。如果 Review 拦截率持续很低（< 20%），可能说明 Review 流于形式，需要改进流程或培训。
